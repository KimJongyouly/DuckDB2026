# 04-1. 숫자, 텍스트, 불리언, 날짜/시간

### 숫자 타입

숫자 타입을 올바르게 선택하면 DuckDB가 연산 비용을 최소화하고 메모리를 효율적으로 사용할 수 있다. 예를 들어 1~100 범위의 상태 코드를 BIGINT로 저장하면 불필요하게 8배의 저장 공간을 소비한다. DuckDB의 숫자 타입은 정수, 부동소수점, 고정밀도 숫자로 구분된다.

```
┌─────────────────────────────────────────────────────────────┐
│                      정수 타입                               │
├──────────────┬────────┬──────────────────────────────────── ┤
│ 타입          │ 크기   │ 범위                                 │
├──────────────┼────────┼──────────────────────────────────── ┤
│ TINYINT      │ 1 byte │ -128 ~ 127                          │
│ SMALLINT     │ 2 byte │ -32,768 ~ 32,767                    │
│ INTEGER      │ 4 byte │ -2,147,483,648 ~ 2,147,483,647      │
│ BIGINT       │ 8 byte │ -9.2×10¹⁸ ~ 9.2×10¹⁸               │
│ HUGEINT      │ 16 byte│ -1.7×10³⁸ ~ 1.7×10³⁸               │
│ UBIGINT      │ 8 byte │ 0 ~ 1.8×10¹⁹ (부호 없음)            │
└──────────────┴────────┴──────────────────────────────────── ┘
```

```sql
-- 정수 리터럴과 타입 확인
SELECT
    42::TINYINT,
    1000::SMALLINT,
    2147483647::INTEGER,
    9999999999::BIGINT,
    typeof(42),            -- 'INTEGER'
    typeof(9999999999);    -- 'BIGINT' (자동 추론)

-- 정수 산술
SELECT 10 / 3;     -- 3 (정수 나눗셈)
SELECT 10 / 3.0;   -- 3.3333... (실수 나눗셈)
SELECT 10 % 3;     -- 1 (나머지)
```

```
┌─────────────────────────────────────────────────────────────┐
│                   부동소수점 타입                             │
├──────────────┬─────────┬───────────────────────────────── ─ ┤
│ 타입          │ 크기    │ 특징                                │
├──────────────┼─────────┼───────────────────────────────── ─ ┤
│ FLOAT        │ 4 byte  │ 단정밀도, 약 7자리 유효숫자          │
│ DOUBLE       │ 8 byte  │ 배정밀도, 약 15자리 유효숫자         │
└──────────────┴─────────┴───────────────────────────────────  ┘
```

```sql
SELECT 3.14::FLOAT;          -- 단정밀도
SELECT 3.141592653589793::DOUBLE;

-- 부동소수점 주의사항
SELECT 0.1 + 0.2 = 0.3;     -- false (부동소수점 오차)
SELECT ABS((0.1 + 0.2) - 0.3) < 1e-10;  -- true (근사 비교)

-- 특수값
SELECT 'inf'::DOUBLE;        -- 양의 무한대
SELECT '-inf'::DOUBLE;       -- 음의 무한대
SELECT 'nan'::DOUBLE;        -- Not a Number
SELECT isnan('nan'::DOUBLE); -- true
SELECT isinf('inf'::DOUBLE); -- true
```

> **주의:** FLOAT과 DOUBLE은 부동소수점 방식으로 값을 저장하기 때문에 `0.1 + 0.2 = 0.3`이 `false`를 반환한다. 금액·수량처럼 정확한 소수점이 필요한 경우에는 FLOAT/DOUBLE 대신 반드시 DECIMAL을 사용해야 한다. 두 실수 값의 동등 비교가 필요하다면 `ABS(a - b) < 허용오차` 형태로 근사 비교를 적용한다.

**DECIMAL / NUMERIC: 고정밀도 숫자**

금액처럼 반올림 오차가 허용되지 않는 값을 다룰 때 DECIMAL을 사용하면 DuckDB가 내부적으로 정수 연산을 통해 정확한 소수점 계산을 보장한다.

```sql
-- DECIMAL(전체 자릿수, 소수점 이하 자릿수)
CREATE TABLE prices (
    product_id INTEGER,
    price DECIMAL(10, 2)  -- 최대 10자리, 소수점 2자리
);

INSERT INTO prices VALUES (1, 9999.99);
INSERT INTO prices VALUES (2, 0.01);

-- DECIMAL은 정확한 연산 보장
SELECT 0.1::DECIMAL(5,1) + 0.2::DECIMAL(5,1);  -- 0.3 (정확)

-- 별칭: NUMERIC, DEC
price NUMERIC(15, 4)
```

> **팁:** DECIMAL(전체 자릿수, 소수점 이하 자릿수)에서 전체 자릿수는 소수점 앞뒤를 모두 합산한 수다. `DECIMAL(5, 2)`는 `999.99`까지 저장할 수 있으며, `1000.00`을 넣으면 오류가 발생한다. 충분한 여유를 두고 전체 자릿수를 지정하는 것이 안전하다.

### 텍스트 타입

문자열 데이터를 다룰 때 DuckDB는 기본적으로 VARCHAR를 권장한다. 길이 제한을 명시하더라도 실제 저장 공간은 입력된 문자열 길이에 따라 동적으로 결정되므로, 굳이 `VARCHAR(n)`을 쓸 이유는 많지 않다.

```
┌────────────────────────────────────────────────────────────┐
│                      텍스트 타입                            │
├────────────┬───────────────────────────────────────────────┤
│ 타입        │ 설명                                          │
├────────────┼───────────────────────────────────────────────┤
│ VARCHAR    │ 가변 길이 문자열 (길이 제한 없음, 권장)          │
│ VARCHAR(n) │ 최대 n자 가변 길이 (n 초과 시 오류)             │
│ TEXT       │ VARCHAR와 동일                                 │
│ CHAR(n)    │ 고정 길이 (n자, 빈 자리는 공백으로 채움)         │
└────────────┴───────────────────────────────────────────────┘
```

```sql
-- 문자열 리터럴
SELECT 'Hello, DuckDB!';
SELECT 'It''s easy';         -- 작은따옴표 이스케이프
SELECT 'line1\nline2';       -- 이스케이프 시퀀스

-- 주요 문자열 함수
SELECT LENGTH('DuckDB');                        -- 6
SELECT UPPER('hello');                          -- 'HELLO'
SELECT LOWER('WORLD');                          -- 'world'
SELECT TRIM('  hello  ');                       -- 'hello'
SELECT LTRIM('  hello');                        -- 'hello'
SELECT RTRIM('hello  ');                        -- 'hello'
SELECT SUBSTRING('DuckDB', 1, 4);              -- 'Duck'
SELECT 'Duck' || 'DB';                          -- 'DuckDB' (연결)
SELECT CONCAT('Duck', 'DB', '!');               -- 'DuckDB!'
SELECT REPLACE('Hello World', 'World', 'DuckDB'); -- 'Hello DuckDB'
SELECT SPLIT_PART('a,b,c', ',', 2);            -- 'b'
SELECT STRING_SPLIT('a,b,c', ',');             -- ['a', 'b', 'c']
SELECT LPAD('42', 5, '0');                     -- '00042'
SELECT RPAD('hi', 5, '.');                     -- 'hi...'

-- 패턴 매칭
SELECT 'DuckDB' LIKE 'Duck%';     -- true (% = 임의 문자열)
SELECT 'DuckDB' LIKE 'D_ckDB';    -- true (_ = 임의 한 문자)
SELECT 'duckdb' ILIKE 'DUCK%';    -- true (대소문자 무시)
SELECT regexp_matches('user@example.com', '^[\w.]+@[\w.]+$');  -- true
SELECT regexp_extract('price: 42.5', '\d+\.?\d*');             -- '42.5'
SELECT regexp_replace('hello world', '\s+', '_');              -- 'hello_world'
```

> **팁:** LIKE는 대소문자를 구분하지만 ILIKE는 구분하지 않는다. 사용자 입력값과 비교할 때 대소문자 처리가 불분명하다면 ILIKE를 쓰거나 `LOWER(col) LIKE 'pattern'` 형태로 통일하면 의도치 않은 누락을 방지할 수 있다.

> **주의:** CHAR(n) 타입은 저장 시 빈 자리를 공백으로 채우기 때문에, `CHAR(5)` 열에 `'abc'`를 넣으면 실제로는 `'abc  '`(공백 2개 포함)로 저장된다. 이 값을 `'abc'`와 비교할 때 DuckDB는 일치로 처리하지만, 다른 시스템에서 가져온 데이터와 섞이면 예상과 다른 결과가 나올 수 있다. 특별한 이유가 없다면 VARCHAR를 사용하는 것이 안전하다.

### 불리언 타입

참/거짓을 나타내는 플래그 열에는 BOOLEAN을 사용하면 DuckDB가 내부적으로 단 1비트로 값을 표현해 공간을 절약하고, `FILTER (WHERE ...)` 절이나 집계 함수와 자연스럽게 연동된다.

```sql
-- BOOLEAN: true / false / NULL
SELECT true, false, NULL::BOOLEAN;

-- 불리언 리터럴 (다양한 표현 허용)
SELECT true, TRUE, 'true'::BOOLEAN, 1::BOOLEAN;   -- 모두 true
SELECT false, FALSE, 'false'::BOOLEAN, 0::BOOLEAN; -- 모두 false

-- 불리언 연산
SELECT true AND false;   -- false
SELECT true OR false;    -- true
SELECT NOT true;         -- false

-- NULL 포함 삼값 논리
SELECT true AND NULL;    -- NULL
SELECT false AND NULL;   -- false  (false AND 무엇 = false)
SELECT true OR NULL;     -- true   (true OR 무엇 = true)
SELECT false OR NULL;    -- NULL

-- 주의: NULL은 false가 아니다
-- WHERE 절에서 NULL은 조건을 만족하지 않는 것으로 처리됨
-- is_active = NULL 이 아니라 is_active IS NULL 로 비교해야 함

-- 집계
SELECT
    COUNT(*) FILTER (WHERE is_active) AS active_cnt,
    BOOL_AND(is_verified) AS all_verified,
    BOOL_OR(is_premium) AS any_premium
FROM users;
```

> **주의:** SQL의 삼값 논리에서 NULL은 "알 수 없음"을 의미한다. `NULL = NULL`은 `true`가 아니라 `NULL`이며, `WHERE col = NULL`은 아무 행도 반환하지 않는다. NULL 비교에는 항상 `IS NULL` 또는 `IS NOT NULL`을 사용해야 한다.

### 날짜/시간 타입

날짜와 시간 데이터를 VARCHAR로 저장하면 정렬, 범위 검색, 기간 계산이 모두 복잡해진다. DATE, TIMESTAMP 타입을 사용하면 DuckDB가 이 모든 연산을 효율적으로 처리하며, 잘못된 날짜 형식을 자동으로 검출해준다.

```
┌───────────────────────────────────────────────────────────────┐
│                     날짜/시간 타입                              │
├─────────────────┬──────────────────────────────────────────── ┤
│ 타입             │ 예시                                         │
├─────────────────┼──────────────────────────────────────────── ┤
│ DATE            │ 2024-03-15                                   │
│ TIME            │ 14:30:00.123456                              │
│ TIMESTAMP       │ 2024-03-15 14:30:00.123456                   │
│ TIMESTAMPTZ     │ 2024-03-15 14:30:00+09:00 (타임존 포함)      │
│ INTERVAL        │ '3 months 15 days 2 hours'                   │
└─────────────────┴──────────────────────────────────────────── ┘
```

```sql
-- 날짜 리터럴
SELECT DATE '2024-03-15';
SELECT TIME '14:30:00';
SELECT TIMESTAMP '2024-03-15 14:30:00';
SELECT TIMESTAMPTZ '2024-03-15 14:30:00+09:00';

-- 현재 날짜/시간
SELECT CURRENT_DATE;         -- 오늘 날짜
SELECT CURRENT_TIME;         -- 현재 시간
SELECT CURRENT_TIMESTAMP;    -- 현재 날짜+시간
SELECT NOW();                -- CURRENT_TIMESTAMP와 동일
SELECT TODAY();              -- CURRENT_DATE와 동일

-- 날짜 부분 추출
SELECT EXTRACT(YEAR FROM DATE '2024-03-15');   -- 2024
SELECT EXTRACT(MONTH FROM DATE '2024-03-15');  -- 3
SELECT EXTRACT(DOW FROM DATE '2024-03-15');    -- 5 (금요일, 0=일요일)
SELECT YEAR(DATE '2024-03-15');    -- 2024 (단축 형태)
SELECT MONTH(DATE '2024-03-15');   -- 3
SELECT DAY(DATE '2024-03-15');     -- 15
SELECT DAYOFWEEK(DATE '2024-03-15');  -- 5
SELECT WEEK(DATE '2024-03-15');    -- 11 (ISO 주차)
SELECT QUARTER(DATE '2024-03-15'); -- 1

-- 날짜 연산
SELECT DATE '2024-03-15' + INTERVAL '30 days';   -- 2024-04-14
SELECT DATE '2024-03-15' - DATE '2024-01-01';     -- 74 (일수)
SELECT DATE '2024-03-15' + INTERVAL '3 months';   -- 2024-06-15
SELECT TIMESTAMPDIFF('day', DATE '2024-01-01', DATE '2024-03-15');  -- 74

-- 날짜 정렬/절사 (TRUNCATE)
SELECT DATE_TRUNC('month', DATE '2024-03-15');    -- 2024-03-01
SELECT DATE_TRUNC('year', DATE '2024-03-15');     -- 2024-01-01
SELECT DATE_TRUNC('week', DATE '2024-03-15');     -- 2024-03-11 (월요일)

-- 포맷팅
SELECT STRFTIME(CURRENT_DATE, '%Y년 %m월 %d일');  -- '2024년 03월 15일'
SELECT STRPTIME('2024-03-15', '%Y-%m-%d');         -- DATE 변환

-- INTERVAL
SELECT INTERVAL '1 year 2 months 15 days';
SELECT INTERVAL '3 hours 30 minutes';
SELECT AGE(DATE '2024-03-15', DATE '1990-06-20');  -- 나이 계산
```

> **팁:** 여러 나라 또는 서버·클라이언트 시간대가 혼용되는 환경에서는 TIMESTAMP 대신 TIMESTAMPTZ를 사용해야 한다. TIMESTAMPTZ는 UTC 기준으로 값을 저장하고, 조회 시 설정된 세션 타임존에 맞춰 자동 변환한다. 이렇게 하면 서머타임(DST) 처리도 DuckDB가 자동으로 수행한다.

> **주의:** `'2024/03/15'::DATE`처럼 슬래시(/) 구분자를 사용하면 DuckDB가 형식을 인식하지 못해 오류를 발생시킨다. 비표준 날짜 형식을 변환할 때는 `STRPTIME('15/03/2024', '%d/%m/%Y')` 함수를 사용해야 한다.
