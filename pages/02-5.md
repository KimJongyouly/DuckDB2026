# 02-5. 동시성 고려사항

DuckDB는 **OLAP 분석 전용**으로 설계된 만큼, 동시성 모델이 MySQL이나 PostgreSQL과 다르다. 기대 수준을 미리 이해하고 사용해야 한다. DuckDB를 다중 사용자 OLTP 데이터베이스처럼 쓰려 한다면 한계에 부딪히지만, 분석 워크로드에서는 이 모델이 오히려 단순하고 예측 가능하다는 장점이 있다.

---

### 기본 원칙: 단일 쓰기, 다중 읽기

DuckDB의 기본 동시성 모델은 간단하다. 핵심만 기억한다면, "한 번에 하나의 프로세스만 쓸 수 있고, 읽기는 여러 프로세스가 동시에 할 수 있다"이다.

| 상황 | 허용 여부 |
|---|---|
| 단일 프로세스, 읽기 | ✅ |
| 단일 프로세스, 쓰기 | ✅ |
| 여러 프로세스, 읽기 전용 | ✅ (`READ_ONLY` 모드) |
| 여러 프로세스, 동시 쓰기 | ❌ 미지원 |

---

### 프로세스 내 동시성 (멀티스레드)

단일 프로세스 안에서 여러 스레드가 DuckDB를 동시에 사용하는 경우를 생각해 보자. 예를 들어, 여러 분석 작업을 병렬로 실행하는 Python 멀티스레딩 코드가 이에 해당한다. 같은 프로세스 안에서 여러 스레드가 동시에 DuckDB를 사용하는 경우, DuckDB는 **MVCC(Multi-Version Concurrency Control)** 와 낙관적 동시성 제어(Optimistic Concurrency Control)를 조합해 충돌을 관리한다.

- 서로 다른 행을 수정하는 트랜잭션은 동시 실행 가능
- `APPEND` 연산은 어떤 경우에도 충돌 없이 동시 수행 가능
- 서로 다른 테이블을 수정하는 경우도 동시 수행 가능
- **같은 행을 동시에 수정하는 경우** → 나중 트랜잭션에서 충돌 에러 발생

```python
# 멀티스레드 환경에서는 반드시 스레드별로 별도 연결 생성
import duckdb
import threading

def worker():
    # ✅ 스레드마다 별도 연결 — 안전
    con = duckdb.connect("my_db.duckdb")
    con.sql("SELECT COUNT(*) FROM my_table").fetchall()
    con.close()

# ❌ 전역 duckdb.sql() 또는 공유 연결을 여러 스레드에서 동시 사용 — 비안전
```

> **주의:** 하나의 연결 객체를 여러 스레드에서 공유하면 충돌 오류가 발생할 수 있다. 스레드당 별도 `duckdb.connect()`를 호출해 독립적인 연결을 유지해야 한다.

### 다중 프로세스 읽기: READ_ONLY 모드

분석 대시보드나 보고서 생성처럼 여러 프로세스가 동일한 DB 파일에서 데이터를 동시에 읽어야 하는 경우, `READ_ONLY` 모드가 해답이다. 여러 프로세스에서 동시에 읽기만 필요하다면 `READ_ONLY` 모드로 연결한다.

```python
# 프로세스 A, B, C 모두 동시에 읽기 가능
con = duckdb.connect("my_database.duckdb", read_only=True)
```

```sql
-- SQL 설정으로도 가능
SET access_mode = 'READ_ONLY';
```

### 다중 프로세스 쓰기: 어떻게 할까?

DuckDB는 여러 프로세스가 동시에 쓰는 시나리오를 기본으로 지원하지 않는다. 이 경우 다음과 같은 우회 전략을 사용한다.

**전략 1: 외부 뮤텍스(mutex) 활용**

프로세스 수준의 잠금을 애플리케이션 코드에서 직접 구현한다.

```python
import duckdb
import multiprocessing

lock = multiprocessing.Lock()

def write_data(data):
    with lock:  # 잠금 획득 후 쓰기
        con = duckdb.connect("shared.duckdb")
        con.execute("INSERT INTO logs VALUES (?)", [data])
        con.close()
```

**전략 2: Parquet / CSV로 분산 저장 후 통합 쿼리**

각 프로세스가 개별 파일에 쓰고, DuckDB가 통합 쿼리 시 파일들을 합쳐 읽는다.

```python
# 각 워커: 개별 Parquet 파일에 저장
df.to_parquet(f"output/worker_{worker_id}.parquet")

# 분석 시: 와일드카드로 모두 읽기
duckdb.sql("SELECT * FROM 'output/*.parquet'")
```

**전략 3: PostgreSQL / MySQL 같은 외부 DB를 쓰기 전용으로 사용**

DuckDB의 확장을 통해 PostgreSQL이나 MySQL에 쓰고, 분석은 DuckDB가 담당하는 아키텍처.

```sql
-- PostgreSQL 확장으로 연결
ATTACH 'host=localhost dbname=app' AS pg (TYPE POSTGRES);

-- PostgreSQL 테이블 읽기
SELECT * FROM pg.orders LIMIT 10;
```

### 동시성 설계 요약

DuckDB는 **고빈도 소규모 트랜잭션(OLTP)** 보다 **대규모 배치 분석(OLAP)** 에 최적화되어 있다. 동시성 모델을 이해하고 용도에 맞게 사용하면 문제가 없다.

| 시나리오 | 권장 방식 |
|---|---|
| 단일 분석 세션 | 기본 연결 하나 |
| 멀티스레드 분석 | 스레드당 별도 연결 |
| 여러 프로세스 읽기 | `READ_ONLY` 모드 |
| 여러 프로세스 쓰기 | 외부 잠금 또는 파일 기반 분산 |
| 고빈도 OLTP 쓰기 | PostgreSQL/MySQL 조합 권장 |
