# 03-2. SELECT 문과 FROM / JOIN

### 기본 SELECT 구조

DuckDB의 SELECT 문은 표준 SQL 순서를 따르며, 실행 순서는 작성 순서와 다르다. 쿼리를 디버깅하거나 최적화할 때 이 실행 순서를 알면 어디서 필터링이 일어나는지, 별칭(alias)을 어느 시점부터 쓸 수 있는지를 정확히 파악할 수 있다.

```
작성 순서          실행 순서
─────────         ─────────
SELECT    →  5    FROM      →  1
FROM      →  1    WHERE     →  2
WHERE     →  2    GROUP BY  →  3
GROUP BY  →  3    HAVING    →  4
HAVING    →  4    SELECT    →  5
ORDER BY  →  6    ORDER BY  →  6
LIMIT     →  7    LIMIT     →  7
```

```sql
-- 기본 구조
SELECT
    department,
    COUNT(*) AS employee_count,
    AVG(salary) AS avg_salary
FROM employees
WHERE hire_date >= '2020-01-01'
GROUP BY department
HAVING COUNT(*) > 5
ORDER BY avg_salary DESC
LIMIT 10;
```

### 열 선택과 표현식

```sql
-- 모든 열 선택
SELECT * FROM employees;

-- 특정 열, 연산, 별칭
SELECT
    first_name || ' ' || last_name AS full_name,
    salary * 1.1 AS new_salary,
    UPPER(department) AS dept
FROM employees;

-- DuckDB 허용: 후행 쉼표
SELECT
    id,
    name,
    salary,      -- 마지막 열 뒤 쉼표 허용
FROM employees;
```

### FROM 절과 테이블 소스

DuckDB의 FROM 절은 단순 테이블 참조 외에도 다양한 소스를 받는다.

```sql
-- 일반 테이블
FROM employees

-- 파일 직접 읽기 (테이블 없이)
FROM 'data/employees.parquet'
FROM read_csv('data/sales.csv')
FROM read_json('data/logs.json')

-- 서브쿼리
FROM (SELECT * FROM employees WHERE active = true) AS active_emp

-- VALUES
FROM (VALUES (1, 'Alice'), (2, 'Bob')) AS t(id, name)
```

### FROM-first 문법 (DuckDB 전용)

데이터를 탐색할 때는 "어떤 테이블에서"가 "어떤 열을"보다 먼저 떠오르는 경우가 많다. DuckDB의 FROM-first 문법은 이 사고 흐름을 그대로 코드로 옮길 수 있게 해준다.

```sql
-- 전통적인 방식
SELECT id, name FROM employees WHERE salary > 50000;

-- FROM-first: 탐색할 때 유용
FROM employees
SELECT id, name
WHERE salary > 50000;
```

> **팁:** FROM-first 문법은 대화형 탐색(interactive exploration) 상황에서 특히 편리하다. 테이블 이름을 먼저 적고 어떤 열이 있는지 확인한 뒤 SELECT 절을 채워나가는 방식으로 쿼리를 점진적으로 작성할 수 있다.

### JOIN 종류

JOIN은 관계형 데이터베이스의 핵심이다. 데이터를 여러 테이블에 나눠 저장해 중복을 줄이는 대신, 필요할 때 JOIN으로 연결해 원하는 형태의 결과를 만든다. 어떤 JOIN을 선택하느냐에 따라 결과 행 수와 NULL 포함 여부가 달라진다.

```sql
-- INNER JOIN: 양쪽에 모두 존재하는 행
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;

-- LEFT JOIN: 왼쪽 테이블 전체 보존
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;

-- FULL OUTER JOIN: 양쪽 전체 보존
SELECT e.name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;

-- CROSS JOIN: 카테시안 곱
SELECT a.item, b.size
FROM products a
CROSS JOIN sizes b;

-- NATURAL JOIN: 동일 이름 열로 자동 조인 (주의해서 사용)
SELECT * FROM employees NATURAL JOIN departments;
```

> **주의:** NATURAL JOIN은 두 테이블에서 이름이 같은 모든 열을 조인 키로 사용한다. 테이블 스키마가 변경되거나 의도하지 않은 열이 같은 이름을 가질 경우 결과가 달라질 수 있으므로, 명시적인 ON 또는 USING 절을 쓰는 편이 안전하다.

### USING과 조인 단순화

```sql
-- ON 대신 USING: 열 이름이 같을 때
SELECT e.name, d.dept_name
FROM employees e
JOIN departments d USING (dept_id);

-- ASOF JOIN: 시계열 데이터에서 가장 가까운 값 매칭 (DuckDB 전용)
SELECT t.ts, t.value, p.price
FROM ticks t
ASOF JOIN prices p ON t.symbol = p.symbol AND t.ts >= p.ts;
```

> **팁:** ASOF JOIN은 금융 틱 데이터, 센서 로그, 이벤트 스트림처럼 타임스탬프 기반 데이터를 다룰 때 특히 유용하다. 이벤트 발생 시각에 가장 가까운 이전 가격이나 상태를 찾는 작업을 별도의 서브쿼리 없이 한 번에 처리한다.
