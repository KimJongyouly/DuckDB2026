# 09-3. Parquet 암호화

DuckDB는 Parquet Modular Encryption(PME) 스펙을 지원한다. 파일 전체 또는 특정 열을 AES-GCM 방식으로 암호화해, 키 없이는 데이터를 읽을 수 없게 한다.

### 암호화 기본 개념

```
┌────────────────────────────────────────────────────────────┐
│             Parquet Modular Encryption                      │
├────────────────────────────────────────────────────────────┤
│  Footer Encryption Key (FEK)                               │
│  → 파일 메타데이터(스키마·통계) 암호화                        │
│  → 이 키 없이는 열 이름조차 알 수 없음                       │
├────────────────────────────────────────────────────────────┤
│  Column Encryption Key (CEK)                               │
│  → 특정 열의 데이터 페이지 암호화                            │
│  → 열마다 다른 키 사용 가능                                  │
│  → FEK와 동일한 키 사용 가능                                 │
├────────────────────────────────────────────────────────────┤
│  암호화 알고리즘: AES-GCM (256비트)                          │
│  무결성 검증: 내장 (데이터 변조 탐지 가능)                    │
└────────────────────────────────────────────────────────────┘
```

### 암호화 쓰기

```sql
-- encryption_config 설정
-- footer_key: 파일 메타데이터 암호화 키
-- column_keys: 열별 암호화 키 (키이름→열목록)

COPY (SELECT * FROM employees) TO 'employees_enc.parquet' (
    FORMAT      PARQUET,
    COMPRESSION ZSTD,
    ENCRYPTION_CONFIG {
        footer_key: 'my_footer_key',
        column_keys: {
            salary_key:  ['salary', 'bonus'],
            personal_key: ['ssn', 'phone', 'email']
        }
    }
);

-- 키 이름은 DuckDB 설정에서 실제 키 값과 매핑
-- (키 값은 반드시 16 또는 32 바이트 길이)
```

### 암호화 키 설정

```sql
-- 방법 1: SET 명령으로 세션에 키 등록
SET secret_key = 'my_footer_key:0123456789abcdef0123456789abcdef';
-- 형식: 'key_name:hex_encoded_key_bytes'

-- 방법 2: parquet_encryption_config 설정
SET parquet_encryption_keys = [
    {
        name:  'my_footer_key',
        key:   '0123456789abcdef0123456789abcdef',  -- 32바이트 hex
        encoding: 'hex'
    },
    {
        name:  'salary_key',
        key:   'fedcba9876543210fedcba9876543210',
        encoding: 'hex'
    },
    {
        name:  'personal_key',
        key:   'aabbccddeeff00112233445566778899',
        encoding: 'hex'
    }
];

-- 방법 3: Python에서 키 설정
```

```python
import duckdb

con = duckdb.connect()

# 키 설정
encryption_config = {
    'footer_key': bytes.fromhex('0123456789abcdef0123456789abcdef'),
    'column_keys': {
        'salary_key':   bytes.fromhex('fedcba9876543210fedcba9876543210'),
        'personal_key': bytes.fromhex('aabbccddeeff00112233445566778899'),
    }
}

# 암호화 쓰기
con.execute("""
    COPY employees TO 'employees_enc.parquet' (
        FORMAT PARQUET,
        ENCRYPTION_CONFIG {
            footer_key: 'main_key',
            column_keys: {
                col_key: ['salary', 'ssn']
            }
        }
    )
""")
```

### 암호화 파일 읽기

```sql
-- 키를 제공해야만 읽기 가능
SELECT * FROM read_parquet(
    'employees_enc.parquet',
    encryption_config = {
        footer_key: 'my_footer_key',
        column_keys: {
            salary_key:   ['salary', 'bonus'],
            personal_key: ['ssn', 'phone', 'email']
        }
    }
);

-- 키 없이 읽으면 오류 발생
SELECT * FROM 'employees_enc.parquet';
-- Error: File is encrypted but no decryption key was provided

-- 잘못된 키로 읽으면 무결성 오류
-- Error: GCM tag mismatch (데이터 변조 또는 잘못된 키)
```

### 부분 열 암호화

민감한 열만 암호화하고 나머지는 평문으로 유지하는 전략이다.

```sql
-- salary, ssn만 암호화, 나머지는 평문
COPY employees TO 'employees_partial.parquet' (
    FORMAT PARQUET,
    ENCRYPTION_CONFIG {
        -- footer_key 없음: 메타데이터는 평문
        column_keys: {
            sensitive_key: ['salary', 'ssn', 'date_of_birth']
        }
    }
);

-- 읽기: 암호화된 열만 키 필요
-- 키 없이 읽기 (암호화된 열은 NULL 또는 오류)
SELECT id, name, department FROM read_parquet('employees_partial.parquet');
-- id | name  | department     (성공: 평문 열)

-- 암호화된 열 읽기 (키 필요)
SELECT salary FROM read_parquet(
    'employees_partial.parquet',
    encryption_config = {column_keys: {sensitive_key: ['salary', 'ssn']}}
);
```

### 암호화 키 관리 실전

```python
import duckdb
import os
import secrets

# 안전한 키 생성 (32바이트 = AES-256)
def generate_key() -> bytes:
    return secrets.token_bytes(32)

# 환경 변수에서 키 로드 (하드코딩 금지)
def load_keys() -> dict:
    return {
        'footer_key': bytes.fromhex(os.environ['PARQUET_FOOTER_KEY']),
        'salary_key': bytes.fromhex(os.environ['PARQUET_SALARY_KEY']),
    }

# 안전한 암호화 쓰기
def write_encrypted(con, table_name: str, output_path: str):
    keys = load_keys()
    # 키를 DuckDB 세션에 등록
    con.execute(f"""
        COPY {table_name} TO '{output_path}' (
            FORMAT PARQUET,
            COMPRESSION ZSTD,
            ENCRYPTION_CONFIG {{
                footer_key: 'footer_key',
                column_keys: {{
                    salary_key: ['salary', 'bonus']
                }}
            }}
        )
    """)

# 안전한 읽기
def read_encrypted(con, path: str):
    keys = load_keys()
    return con.execute(f"""
        SELECT * FROM read_parquet(
            '{path}',
            encryption_config = {{
                footer_key: 'footer_key',
                column_keys: {{
                    salary_key: ['salary', 'bonus']
                }}
            }}
        )
    """).df()
```

### 암호화 주의사항

```
┌────────────────────────────────────────────────────────────┐
│                  암호화 사용 시 주의사항                      │
├────────────────────────────────────────────────────────────┤
│  1. 키는 코드에 하드코딩 금지                                │
│     → 환경 변수, AWS KMS, HashiCorp Vault 등 활용           │
│                                                            │
│  2. 키를 잃으면 데이터도 잃는다                              │
│     → 키 백업과 로테이션 정책 필수                           │
│                                                            │
│  3. footer_key 암호화 시 통계도 암호화됨                     │
│     → Zone Map 필터링 효과 없어짐 (성능 저하 가능)           │
│                                                            │
│  4. 열별 다른 키 사용 시 접근 제어 가능                      │
│     → 팀/역할별로 다른 키 배포                              │
│                                                            │
│  5. AES-GCM은 무결성 검증 내장                              │
│     → 데이터 변조 시 읽기 오류로 탐지                        │
└────────────────────────────────────────────────────────────┘
```

### Parquet 최적화 전략 종합

```sql
-- 최적의 Parquet 파일 생성 체크리스트

-- 1. 압축: ZSTD (균형), SNAPPY (속도 우선)
COPY data TO 'out.parquet' (FORMAT PARQUET, COMPRESSION ZSTD);

-- 2. Row Group 크기: 50만~100만 행 (128MB~512MB)
COPY data TO 'out.parquet' (FORMAT PARQUET, ROW_GROUP_SIZE 500000);

-- 3. 파티션: 자주 필터링하는 열 기준
COPY data TO 'out/' (FORMAT PARQUET, PARTITION_BY (year, region));

-- 4. 정렬: 범위 쿼리가 많은 열은 정렬 후 저장 (Zone Map 효과 극대화)
COPY (SELECT * FROM events ORDER BY ts) TO 'events_sorted.parquet'
(FORMAT PARQUET, COMPRESSION ZSTD);

-- 5. 열 선택: 불필요한 열 제외
COPY (SELECT id, ts, event_type, user_id FROM events)
TO 'events_slim.parquet' (FORMAT PARQUET);

-- 6. 암호화: 민감 열만 부분 암호화 (성능과 보안 균형)
COPY data TO 'data_enc.parquet' (
    FORMAT PARQUET,
    COMPRESSION ZSTD,
    ENCRYPTION_CONFIG {column_keys: {pii_key: ['ssn', 'email', 'phone']}}
);
```
