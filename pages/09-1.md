# 09-1. Parquet 읽기·쓰기

### Parquet 구조 이해

DuckDB가 Parquet 파일을 효율적으로 처리하는 이유를 이해하려면, 파일 내부가 어떻게 구성되어 있는지 먼저 살펴볼 필요가 있다. Parquet 파일은 크게 헤더·Row Group·푸터 세 부분으로 나뉘며, 이 구조 덕분에 DuckDB는 필요한 열과 행 범위만 선택적으로 읽을 수 있다.

```
┌────────────────────────────────────────────────────────────┐
│                  Parquet 파일 구조                          │
├────────────────────────────────────────────────────────────┤
│  File Header (매직 바이트: PAR1)                            │
├──────────────────────────┬─────────────────────────────── ┤
│  Row Group 0             │  Row Group 1                    │
│  ┌──────┬──────┬──────┐  │  ┌──────┬──────┬──────┐        │
│  │ Col0 │ Col1 │ Col2 │  │  │ Col0 │ Col1 │ Col2 │        │
│  │ Page │ Page │ Page │  │  │ Page │ Page │ Page │        │
│  └──────┴──────┴──────┘  │  └──────┴──────┴──────┘        │
├────────────────────────────────────────────────────────────┤
│  File Footer (스키마 + 통계 메타데이터)                      │
│  - 각 Row Group의 오프셋                                    │
│  - 각 열의 min/max/null_count                              │
│  - 압축 코덱, 인코딩 정보                                   │
├────────────────────────────────────────────────────────────┤
│  File Footer Length + 매직 바이트 (PAR1)                    │
└────────────────────────────────────────────────────────────┘
```

핵심 개념을 정리하면 다음과 같다. **Row Group**은 수십만~수백만 행 단위의 데이터 덩어리로, 각 Row Group 안에서 데이터는 열 단위로 저장된다. **File Footer**는 파일 맨 끝에 위치하며, 각 Row Group과 열의 오프셋·min/max 통계·압축 정보를 모두 담고 있다. DuckDB는 쿼리를 실행할 때 Footer를 먼저 읽고, 필요한 Row Group과 열만 골라서 읽는다. 이 과정에서 불필요한 I/O가 대폭 줄어든다.

> **팁:** 파일 끝의 Footer를 먼저 읽어야 하기 때문에, Parquet를 S3 등 원격 스토리지에서 읽을 때는 네트워크 왕복 횟수를 최소화하는 것이 중요하다. DuckDB는 내부적으로 Footer 캐싱을 통해 이를 최적화한다.

```
CSV vs Parquet 비교
────────────────────────────────────────────────────────
항목           CSV              Parquet
─────────────  ──────────────   ──────────────────────
저장 방식      행 지향          열 지향
타입 정보      없음 (추론)      내장
압축           없음/외부        내장 (Snappy/ZSTD/GZIP)
스키마         없음             내장
통계           없음             min/max/null_count 내장
선택적 읽기    열 선택 불가     열 단위 스킵 가능
크기           크다             1/3~1/10 수준
쿼리 속도      느림             수십 배 빠름
사람이 읽기    가능             불가 (바이너리)
```

### 가장 간단한 읽기

구조를 이해했다면 실제로 읽어보자. DuckDB는 파일 경로를 `FROM` 절에 직접 넣는 것만으로 Parquet 파일을 쿼리한다. 별도의 임포트 단계나 테이블 생성이 필요 없다.

```sql
-- 파일을 직접 쿼리
SELECT * FROM 'data.parquet';
SELECT * FROM 'warehouse/sales/2024.parquet' LIMIT 100;

-- read_parquet 함수
SELECT * FROM read_parquet('data.parquet');

-- 기본 정보 확인
DESCRIBE SELECT * FROM 'data.parquet';
SELECT COUNT(*) FROM 'data.parquet';
```

### read_parquet 주요 옵션

`read_parquet()` 함수는 단순 읽기를 넘어 다양한 옵션을 제공한다. 특히 열 선택(Projection Pushdown)과 Hive 파티션 인식은 대용량 파일에서 읽는 데이터 양을 직접적으로 줄여 준다.

```sql
SELECT * FROM read_parquet(
    'sales.parquet',

    -- 열 선택 (Projection Pushdown)
    -- 필요한 열만 읽어 I/O 최소화
    columns = ['id', 'date', 'amount', 'region'],

    -- 파티션 인식
    hive_partitioning = true,   -- 디렉터리명에서 파티션 열 추출

    -- 파일 출처 추적
    filename = true,

    -- 스키마 통합 (여러 파일 스키마가 다를 때)
    union_by_name = true,

    -- 파일 행 번호 (디버깅용)
    file_row_number = true
);
```

### 여러 파일 읽기

실무에서는 데이터가 단일 파일이 아닌 여러 파일에 분산되어 있는 경우가 많다. DuckDB는 와일드카드와 배열 표기를 통해 여러 파일을 하나의 논리적 테이블처럼 처리한다.

```sql
-- 와일드카드
SELECT * FROM 'data/*.parquet';
SELECT * FROM read_parquet('logs/2024_*.parquet');

-- 배열로 지정
SELECT * FROM read_parquet(['jan.parquet', 'feb.parquet', 'mar.parquet']);

-- 재귀 와일드카드 (하위 디렉터리 포함)
SELECT * FROM 'warehouse/**/*.parquet';

-- 파일 출처 추적
SELECT filename, COUNT(*) AS rows
FROM read_parquet('data/*.parquet', filename=true)
GROUP BY filename;
```

### Hive 파티션 읽기

데이터 레이크나 데이터 웨어하우스에서는 디렉터리 구조 자체가 파티션 정보를 담는 Hive 스타일이 널리 쓰인다. DuckDB는 `hive_partitioning=true` 옵션 하나로 이 디렉터리 구조를 자동으로 해석해, 파티션 열을 쿼리 결과에 추가하고 필터 조건에 맞는 디렉터리만 선택적으로 읽는다.

Hive 스타일 파티션은 디렉터리 이름에 `key=value` 형식으로 파티션 정보를 담는다.

```
warehouse/
├── year=2023/
│   ├── region=Seoul/
│   │   └── data.parquet
│   └── region=Busan/
│       └── data.parquet
└── year=2024/
    ├── region=Seoul/
    │   └── data.parquet
    └── region=Busan/
        └── data.parquet
```

```sql
-- hive_partitioning=true: 디렉터리명이 열로 자동 추가
SELECT *
FROM read_parquet('warehouse/**/*.parquet', hive_partitioning=true);
-- year | region | id | amount | ...
-- ─────┼────────┼────┼────────┼───
-- 2023 | Seoul  |  1 |  15000 | ...
-- 2023 | Busan  |  2 |   8500 | ...

-- 파티션 열로 필터링 (Partition Pruning: 해당 디렉터리만 읽음)
SELECT *
FROM read_parquet('warehouse/**/*.parquet', hive_partitioning=true)
WHERE year = 2024 AND region = 'Seoul';
-- year=2024/region=Seoul/ 디렉터리만 실제로 읽음

-- 파티션 타입 명시
SELECT *
FROM read_parquet(
    'warehouse/**/*.parquet',
    hive_partitioning = true,
    hive_types = {'year': 'INTEGER', 'region': 'VARCHAR'}
);
```

> **팁:** `WHERE year = 2024 AND region = 'Seoul'` 처럼 파티션 열로 필터링하면, DuckDB는 해당 조건에 맞는 디렉터리만 실제로 읽는다(Partition Pruning). 파티션 설계를 잘 하면 쿼리 범위를 물리적으로 좁힐 수 있다.

### 스키마가 다른 파일 통합

시간이 지나면서 열이 추가되거나 변경된 파일들을 함께 쿼리해야 할 때가 있다. `union_by_name=true`를 지정하면 DuckDB가 열 이름 기준으로 스키마를 통합하고, 특정 파일에 없는 열은 자동으로 NULL로 채운다.

```sql
-- union_by_name: 열 이름으로 매칭, 없는 열은 NULL
-- (파일마다 열 수나 순서가 다를 때)
SELECT *
FROM read_parquet('data/*.parquet', union_by_name=true);

-- 어느 파일에서 왔는지 + 스키마 통합
SELECT filename, *
FROM read_parquet(
    'data/*.parquet',
    filename=true,
    union_by_name=true
);
```

### Parquet 쓰기

```sql
-- 기본 내보내기
COPY employees TO 'employees.parquet' (FORMAT PARQUET);

-- 쿼리 결과 내보내기
COPY (
    SELECT id, name, department, salary
    FROM employees
    WHERE active = true
) TO 'active_employees.parquet' (FORMAT PARQUET);

-- 압축 코덱 선택
COPY employees TO 'employees.parquet' (
    FORMAT      PARQUET,
    COMPRESSION ZSTD,       -- SNAPPY(기본), GZIP, ZSTD, BROTLI, LZ4, UNCOMPRESSED
    ROW_GROUP_SIZE 100000   -- Row Group 크기 (행 수)
);
```

```
압축 코덱 비교
───────────────────────────────────────────────────────
코덱        압축률    속도    용도
──────────  ────────  ──────  ─────────────────────────
SNAPPY      보통      매우빠름  기본값, 실시간 처리
ZSTD        높음      빠름    균형 잡힌 선택 (권장)
GZIP        높음      느림    장기 보관, S3 비용 절감
LZ4         낮음      최고속  초저지연 처리
UNCOMPRESSED 없음     최고속  임시 파일, 디버깅
```

### Hive 파티션으로 쓰기

```sql
-- PARTITION_BY로 Hive 스타일 파티션 생성
COPY (SELECT * FROM sales) TO 'warehouse/' (
    FORMAT       PARQUET,
    PARTITION_BY (year, region),
    COMPRESSION  ZSTD,
    OVERWRITE_OR_IGNORE true
);
-- 결과:
-- warehouse/year=2023/region=Seoul/data_0.parquet
-- warehouse/year=2023/region=Busan/data_0.parquet
-- warehouse/year=2024/region=Seoul/data_0.parquet

-- 파생 파티션 열 추가 후 쓰기
COPY (
    SELECT
        *,
        YEAR(order_date)    AS year,
        MONTH(order_date)   AS month
    FROM orders
) TO 'orders_partitioned/' (
    FORMAT       PARQUET,
    PARTITION_BY (year, month),
    COMPRESSION  ZSTD
);
```

### Python에서 Parquet 처리

```python
import duckdb

con = duckdb.connect()

# 읽기: DataFrame으로
df = con.execute("SELECT * FROM 'data.parquet'").df()

# 읽기: Arrow Table로 (대용량에서 효율적)
import pyarrow.parquet as pq
arrow_table = con.execute("SELECT * FROM 'data.parquet'").arrow()

# Pandas DataFrame → Parquet
import pandas as pd
df = pd.read_parquet('input.parquet')
con.register('df', df)
con.execute("COPY df TO 'output.parquet' (FORMAT PARQUET, COMPRESSION ZSTD)")

# PyArrow Table → Parquet
import pyarrow as pa
table = pa.table({'x': [1, 2, 3], 'y': ['a', 'b', 'c']})
con.register('tbl', table)
con.execute("COPY tbl TO 'arrow_out.parquet' (FORMAT PARQUET)")

# 스트리밍: 대용량 파일 배치 처리
rel = con.read_parquet('huge.parquet')
for batch in rel.fetchmany(50000):
    process(batch)
```
