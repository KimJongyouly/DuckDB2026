# 09-2. 메타데이터 활용

Parquet의 강력한 특징 중 하나는 파일 내부에 풍부한 메타데이터를 담는다는 점이다. DuckDB는 이 메타데이터를 쿼리로 조회할 수 있다.

### 파일 메타데이터 조회

```sql
-- 기본 메타데이터
SELECT * FROM parquet_metadata('sales.parquet');
```

`parquet_metadata`가 반환하는 주요 열:

```
┌──────────────────────┬────────────────────────────────────────────┐
│ 열 이름               │ 설명                                        │
├──────────────────────┼────────────────────────────────────────────┤
│ file_name            │ 파일 경로                                   │
│ row_group_id         │ Row Group 번호 (0-based)                    │
│ row_group_num_rows   │ 해당 Row Group의 행 수                      │
│ row_group_bytes      │ 해당 Row Group의 압축 후 크기               │
│ column_id            │ 열 번호                                     │
│ file_offset          │ 파일 내 열 데이터 오프셋                     │
│ num_values           │ 값의 수 (NULL 포함)                         │
│ compression          │ 압축 코덱 (SNAPPY, ZSTD 등)                │
│ encodings            │ 인코딩 방식 목록                            │
│ index_page_offset    │ 인덱스 페이지 오프셋                         │
│ dictionary_page_offset│ 딕셔너리 페이지 오프셋                      │
│ data_page_offset     │ 데이터 페이지 오프셋                         │
│ total_compressed_size│ 압축 후 크기                                │
│ total_uncompressed_size│ 압축 전 크기                              │
└──────────────────────┴────────────────────────────────────────────┘
```

### 스키마 메타데이터

```sql
-- 파일 스키마 확인
SELECT * FROM parquet_schema('sales.parquet');
-- file_name | name        | type    | type_length | ...
-- ──────────┼─────────────┼─────────┼─────────────┼───
-- sales.pq  | id          | INT32   |             |
-- sales.pq  | order_date  | INT32   | (DATE)      |
-- sales.pq  | amount      | DOUBLE  |             |
-- sales.pq  | region      | BYTE_ARRAY (UTF8)     |

-- 여러 파일 스키마 비교
SELECT file_name, name, type
FROM parquet_schema('data/*.parquet')
ORDER BY file_name, name;
```

### 열 통계 활용 (Zone Map / Min-Max 필터)

Parquet는 각 Row Group의 각 열에 대해 `min`, `max`, `null_count`를 저장한다. DuckDB는 이 정보를 활용해 필요 없는 Row Group을 아예 읽지 않는다(Zone Map 필터링).

```sql
-- 열 통계 조회
SELECT * FROM parquet_file_metadata('sales.parquet');

-- Row Group별 열 통계
SELECT
    row_group_id,
    column_id,
    stats_min,
    stats_max,
    stats_null_count,
    stats_distinct_count
FROM parquet_metadata('sales.parquet')
WHERE column_id = 2  -- amount 열
ORDER BY row_group_id;
```

```
Zone Map 필터링 동작 예시
───────────────────────────────────────────────────────────
쿼리: WHERE amount > 50000

Row Group 0: amount min=100,    max=45000  → 스킵 (max < 50000)
Row Group 1: amount min=1000,   max=98000  → 읽음 (max >= 50000)
Row Group 2: amount min=50001,  max=120000 → 읽음 (min > 50000)
Row Group 3: amount min=500,    max=49999  → 스킵 (max < 50000)

→ 4개 Row Group 중 2개만 실제로 읽음 (50% I/O 절감)
```

```sql
-- Zone Map이 효과적인 쿼리 패턴
-- ✓ 범위 필터 (정렬된 열에 효과적)
SELECT * FROM 'sales.parquet' WHERE amount BETWEEN 10000 AND 50000;
SELECT * FROM 'events.parquet' WHERE ts >= '2024-01-01';

-- ✓ 등치 필터
SELECT * FROM 'sales.parquet' WHERE region = 'Seoul';

-- Zone Map이 덜 효과적인 패턴
-- ✗ 열 변환 후 필터 (통계 활용 불가)
SELECT * FROM 'sales.parquet' WHERE YEAR(ts) = 2024;
-- 대신:
SELECT * FROM 'sales.parquet' WHERE ts >= '2024-01-01' AND ts < '2025-01-01';
```

### 메타데이터로 파일 분석

```sql
-- 파일 크기와 압축률 분석
SELECT
    file_name,
    SUM(row_group_num_rows)          AS total_rows,
    SUM(total_compressed_size)       AS compressed_bytes,
    SUM(total_uncompressed_size)     AS uncompressed_bytes,
    ROUND(
        SUM(total_compressed_size) * 100.0
        / SUM(total_uncompressed_size), 1
    )                                AS compression_pct,
    COUNT(DISTINCT row_group_id)     AS row_groups
FROM parquet_metadata('data/*.parquet')
GROUP BY file_name
ORDER BY compressed_bytes DESC;

-- 열별 압축률 비교
SELECT
    column_id,
    SUM(total_compressed_size)   AS compressed,
    SUM(total_uncompressed_size) AS uncompressed,
    ROUND(
        100.0 - SUM(total_compressed_size) * 100.0
        / SUM(total_uncompressed_size), 1
    ) AS saved_pct
FROM parquet_metadata('sales.parquet')
GROUP BY column_id
ORDER BY saved_pct DESC;

-- Row Group 크기 분포 확인 (최적화 판단)
SELECT
    row_group_id,
    row_group_num_rows,
    ROUND(row_group_bytes / 1024.0 / 1024.0, 2) AS size_mb
FROM parquet_metadata('sales.parquet')
ORDER BY row_group_id;
```

### 커스텀 메타데이터 (Key-Value Metadata)

```sql
-- 파일 수준 커스텀 메타데이터 확인
SELECT * FROM parquet_kv_metadata('sales.parquet');
-- file_name   | key           | value
-- ────────────┼───────────────┼───────────────────
-- sales.pq    | created_by    | DuckDB v1.1.0
-- sales.pq    | pandas        | {"index_columns":...}
-- sales.pq    | arrow:schema  | <Arrow IPC schema>

-- pandas 메타데이터로 원래 타입 정보 복원 가능
```

### 통계 기반 파일 최적화 판단

```sql
-- 파일 분할 필요성 판단
-- Row Group이 너무 크거나(>500MB) 너무 작으면(<10MB) 재구성 고려
WITH rg_stats AS (
    SELECT
        file_name,
        row_group_id,
        row_group_num_rows,
        row_group_bytes / 1024.0 / 1024.0 AS size_mb
    FROM parquet_metadata('data/*.parquet')
)
SELECT
    file_name,
    COUNT(*) AS num_row_groups,
    AVG(row_group_num_rows) AS avg_rows_per_rg,
    AVG(size_mb) AS avg_rg_mb,
    MIN(size_mb) AS min_rg_mb,
    MAX(size_mb) AS max_rg_mb,
    CASE
        WHEN AVG(size_mb) < 10  THEN '너무 작음 → 병합 고려'
        WHEN AVG(size_mb) > 500 THEN '너무 큼  → 분할 고려'
        ELSE '적정 크기'
    END AS recommendation
FROM rg_stats
GROUP BY file_name;
```
