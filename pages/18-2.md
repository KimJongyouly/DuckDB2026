# 18-2. 병렬 처리와 메모리 관리

앞서 실행 계획을 통해 쿼리가 어떤 단계를 거치는지 파악했다면, 이번에는 각 단계가 CPU와 메모리를 어떻게 활용하는지를 살펴본다. DuckDB가 왜 빠른지 그 핵심 원리를 이해하면, 설정 몇 가지만으로도 성능을 크게 끌어올릴 수 있다.

DuckDB는 벡터화 실행 엔진과 멀티스레드 병렬 처리를 기본으로 지원한다. 하드웨어 자원을 최대한 활용하는 방법을 이해하면 대용량 쿼리 성능을 크게 향상시킬 수 있다.

### 벡터화 실행 엔진 원리

DuckDB가 빠른 핵심 이유 중 하나는 **벡터화 실행 엔진**이다. 이를 쉽게 설명하면, CPU가 한 번에 여러 값을 동시에 계산하는 방식이라고 이해하면 된다. 전통적인 방식이 계산기로 숫자를 한 개씩 입력하는 것이라면, 벡터화는 엑셀 수식처럼 한 번에 열 전체에 연산을 적용하는 것과 같다. 내부적으로는 SIMD(Single Instruction Multiple Data)라는 CPU 기능을 활용해, 하나의 명령으로 수십 개의 값을 동시에 처리한다.

```
전통적 행 단위 처리:
  row1 → filter → project → aggregate
  row2 → filter → project → aggregate
  ...  (각 행마다 함수 호출 오버헤드 발생)

DuckDB 벡터 처리 (VECTOR_SIZE = 2048행 단위):
  [row1..row2048] → SIMD filter → SIMD project → SIMD aggregate
  [row2049..row4096] → ...
  (CPU 캐시 활용 + SIMD 명령어로 병렬 계산)
```

벡터 크기는 기본 2048행이며, 내부적으로 SIMD(AVX-512 등) 명령어를 활용해 한 번의 CPU 명령으로 여러 값을 동시에 처리한다.

### 병렬 처리 설정

```sql
-- 현재 사용 중인 스레드 수 확인
SELECT current_setting('threads');
PRAGMA threads;

-- 스레드 수 조정 (기본: CPU 논리 코어 수)
SET threads = 8;     -- 8스레드로 제한
SET threads = 1;     -- 단일 스레드 (디버깅용)
RESET threads;       -- CPU 코어 수로 복원

-- 코어 수 확인 (Linux)
-- nproc 또는 lscpu
```

```python
import duckdb
import os

# CPU 코어 수 자동 설정
cpu_count = os.cpu_count()
con = duckdb.connect(config={
    'threads': cpu_count,
    'memory_limit': f'{int(os.popen("free -g").readlines()[1].split()[1]) * 0.8:.0f}GB'
})

print(f"스레드: {con.execute('SELECT current_setting($$threads$$)').fetchone()[0]}")
print(f"메모리: {con.execute('SELECT current_setting($$memory_limit$$)').fetchone()[0]}")
```

### 메모리 설정 최적화

```sql
-- 메모리 한도 설정 (기본: RAM의 약 80%)
SET memory_limit = '32GB';

-- 스필 디렉터리 설정 (SSD 권장)
SET temp_directory = '/fast_ssd/duckdb_spill';

-- 메모리 사용 현황 조회
SELECT
    memory_usage,
    memory_limit,
    round(memory_usage / memory_limit * 100, 1) AS usage_pct
FROM duckdb_memory();
```

```python
# 환경별 메모리 설정 패턴
import psutil
import duckdb

total_ram = psutil.virtual_memory().total
available_ram = psutil.virtual_memory().available

# 전체 RAM의 75%를 한도로 설정
memory_limit_gb = int(total_ram * 0.75 / (1024**3))

con = duckdb.connect(config={
    'memory_limit': f'{memory_limit_gb}GB',
    'temp_directory': '/tmp/duckdb',
    'threads': psutil.cpu_count(logical=False),  # 물리 코어만
})

print(f"메모리 한도: {memory_limit_gb}GB / 전체 RAM: {total_ram / (1024**3):.0f}GB")
```

### 메모리 효율적인 쿼리 패턴

```sql
-- 나쁜 예: 불필요한 컬럼까지 모두 읽음
SELECT *
FROM large_table
WHERE status = 'active';

-- 좋은 예: 필요한 컬럼만 선택 (Projection Pushdown 명시)
SELECT id, name, email
FROM large_table
WHERE status = 'active';

-- 나쁜 예: 대용량 데이터를 메모리에 올린 후 필터
SELECT *
FROM (SELECT * FROM sales) sub
WHERE year = 2024;

-- 좋은 예: 필터를 가능한 한 일찍 적용
SELECT region, SUM(amount)
FROM sales
WHERE year = 2024
  AND status = 'completed'
GROUP BY region;

-- 나쁜 예: DISTINCT를 무분별하게 사용 (정렬 비용 발생)
SELECT DISTINCT customer_id FROM orders;

-- 좋은 예: GROUP BY로 대체 (해시 집계, 더 효율적)
SELECT customer_id FROM orders GROUP BY customer_id;
```

### 해시 조인 최적화

DuckDB는 조인의 두 테이블 중 작은 쪽을 해시 테이블로 빌드하고, 큰 쪽을 프로브한다. 작은 테이블이 메모리에 들어오지 않으면 스필이 발생한다.

```sql
-- 작은 차원 테이블을 항상 오른쪽(빌드 측)에 배치
-- DuckDB가 자동으로 판단하지만, 통계가 부정확할 때는 순서가 영향을 줌

-- 조인 전에 필터를 적용해 빌드 측 크기를 줄임
SELECT o.order_id, c.region, o.amount
FROM orders o
JOIN (
    SELECT id, region
    FROM customers
    WHERE region IN ('Asia', 'Europe')  -- 먼저 필터링
) c ON o.customer_id = c.id
WHERE o.order_date >= '2024-01-01';

-- 조인 순서 힌트 (자동 최적화가 잘못됐을 때)
-- DuckDB는 join_order_optimization으로 자동 재배열
SET join_order_optimization = true;  -- 기본값
```

### 집계 최적화

```sql
-- GROUP BY ALL: 모든 비집계 컬럼 자동 포함
SELECT year, region, product,
       SUM(amount), COUNT(*)
FROM sales
GROUP BY ALL;   -- year, region, product 자동 추론

-- FILTER 절: 여러 집계를 한 번의 스캔으로
SELECT
    COUNT(*) FILTER (WHERE status = 'completed') AS completed,
    COUNT(*) FILTER (WHERE status = 'pending')   AS pending,
    SUM(amount) FILTER (WHERE year = 2024)       AS revenue_2024,
    SUM(amount) FILTER (WHERE year = 2023)       AS revenue_2023
FROM orders;
-- 위 쿼리는 테이블을 한 번만 스캔

-- APPROXIMATE: 정확도보다 속도가 중요할 때
SELECT
    APPROX_COUNT_DISTINCT(user_id) AS approx_unique_users,
    COUNT(DISTINCT user_id)        AS exact_unique_users
FROM events;
```
