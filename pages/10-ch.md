# 제10장: 다중 파일과 파티셔닝

실무 데이터 파이프라인은 단일 파일이 아니라 수백, 수천 개의 파일로 구성된다. 날짜별로 쌓인 로그, 지역별로 분리된 판매 데이터, 월별로 나뉜 거래 내역이 그 예다. DuckDB는 이런 파일 모음을 단일 테이블처럼 쿼리하고, 결과를 다시 파티션 구조로 저장하는 일련의 워크플로우를 SQL 한 줄로 처리한다. 이 장에서는 다중 파일 읽기부터 파티션 쓰기까지의 전 과정을 체계적으로 살펴본다.

```sql
-- 와일드카드로 여러 파일 읽기
SELECT * FROM 'data/*.parquet';
SELECT * FROM 'warehouse/**/*.parquet';

-- 스키마가 다른 파일 통합
SELECT * FROM read_parquet('data/*.parquet', union_by_name=true);

-- Hive 파티션으로 쓰기
COPY (SELECT * FROM sales) TO 'warehouse/' (
    FORMAT       PARQUET,
    PARTITION_BY (year, month, region),
    COMPRESSION  ZSTD
);
```

## 주요 학습 내용

- 와일드카드(`*.parquet`, `**/*.parquet`)와 `union_by_name=true`를 조합하면 열 순서나 구조가 다른 파일들도 안전하게 통합할 수 있다.
- Partition Pruning은 WHERE 절에 파티션 열(연도, 월 등)을 포함시키는 것만으로 활성화된다. 수백 개의 파일 중 조건에 맞는 소수만 읽어 I/O를 극적으로 줄인다.
- 파티션당 파일 크기는 128MB~512MB가 적정하다. 그보다 작은 소형 파일이 수백 개 쌓이면 파일 열기 비용이 누적되어 성능이 저하되므로 주기적으로 병합해야 한다.
