# 18-1. 실행 계획(EXPLAIN) 읽기

실행 계획은 DuckDB가 쿼리를 어떤 순서로, 어떤 연산자를 통해 처리하는지 보여주는 지도다. 느린 쿼리를 최적화할 때 가장 먼저 확인해야 할 정보가 바로 여기에 있다. 마치 요리 레시피처럼, 어떤 재료(데이터)를 어떤 순서로 가공하는지 한눈에 파악할 수 있다.

### EXPLAIN vs EXPLAIN ANALYZE

```sql
-- EXPLAIN: 실행하지 않고 계획만 출력
EXPLAIN
SELECT region, SUM(amount) AS total
FROM sales
WHERE year = 2024
GROUP BY region
ORDER BY total DESC;

-- EXPLAIN ANALYZE: 실제 실행 후 실측값 포함
EXPLAIN ANALYZE
SELECT region, SUM(amount) AS total
FROM sales
WHERE year = 2024
GROUP BY region
ORDER BY total DESC;
```

`EXPLAIN`은 빠르게 계획을 확인할 때, `EXPLAIN ANALYZE`는 실제 행 수·소요 시간·메모리 사용량까지 확인할 때 사용한다.

### 실행 계획 읽는 법

아래 다이어그램은 처음 보면 조금 낯설게 느껴질 수 있지만, 규칙은 하나다. **가장 아래 박스에서 시작해서 위로 올라가며 읽는다.** 데이터가 흐르는 방향을 따라가면 된다.

**단계별 읽기 순서:**

1. **가장 아래 (SEQ_SCAN)**: 테이블에서 데이터를 읽는 출발점이다. 여기서 몇 행을 읽는지 먼저 확인한다.
2. **FILTER**: 읽은 데이터 중 WHERE 조건을 만족하는 행만 남긴다. 이 단계에서 행 수가 크게 줄어드는지 확인하자.
3. **HASH_GROUP_BY**: 남은 데이터를 묶어 집계한다.
4. **ORDER BY**: 결과를 정렬한다.
5. **가장 위 (RESULT)**: 최종 결과가 출력된다.

```
┌───────────────────────────────────────────────────────────────────┐
│                     RESULT (물리 계획 최상단)                      │
│  출력: 최종 결과                                                    │
└─────────────────────────┬─────────────────────────────────────────┘
                          │
┌─────────────────────────┴─────────────────────────────────────────┐
│                  ORDER BY (정렬)                                    │
│  total DESC                                                        │
└─────────────────────────┬─────────────────────────────────────────┘
                          │
┌─────────────────────────┴─────────────────────────────────────────┐
│              HASH_GROUP_BY (집계)                                   │
│  Groups: region                                                    │
│  Aggregates: sum(amount)                                           │
│  ~50 Rows (추정)                                                   │
└─────────────────────────┬─────────────────────────────────────────┘
                          │
┌─────────────────────────┴─────────────────────────────────────────┐
│              FILTER (필터 푸시다운)                                  │
│  year = 2024                                                       │
└─────────────────────────┬─────────────────────────────────────────┘
                          │
┌─────────────────────────┴─────────────────────────────────────────┐
│              SEQ_SCAN (순차 스캔)                                   │
│  Table: sales                                                      │
│  Projections: region, amount, year                                 │
│  ~10,000,000 Rows                                                  │
└───────────────────────────────────────────────────────────────────┘
```

계획은 **아래에서 위로** 읽는다. 데이터는 가장 아래의 스캔에서 시작해 위로 올라가며 변환된다. 각 단계에서 행 수가 얼마나 줄어드는지 주목하면, 어느 단계가 가장 많은 일을 하는지 금방 알 수 있다.

### 주요 연산자 해설

```
┌────────────────────────┬──────────────────────────────────────────────────────┐
│ 연산자                  │ 의미                                                  │
├────────────────────────┼──────────────────────────────────────────────────────┤
│ SEQ_SCAN               │ 테이블 전체 순차 스캔                                  │
│ INDEX_SCAN             │ ART 인덱스를 통한 포인트 룩업                          │
│ FILTER                 │ WHERE 조건 필터링                                     │
│ PROJECTION             │ SELECT 컬럼 선택 (불필요한 컬럼 제거)                  │
│ HASH_JOIN              │ 해시 기반 조인 (대용량 테이블 간)                      │
│ NESTED_LOOP_JOIN       │ 중첩 루프 조인 (소규모 또는 인덱스 조인)               │
│ HASH_GROUP_BY          │ 해시 기반 GROUP BY                                    │
│ PERFECT_HASH_GROUP_BY  │ 카디널리티가 낮을 때 사용하는 최적화된 집계             │
│ SORT                   │ ORDER BY 정렬                                         │
│ TOP_N                  │ ORDER BY + LIMIT 통합 최적화 (전체 정렬 불필요)        │
│ LIMIT                  │ LIMIT / OFFSET                                        │
│ WINDOW                 │ 윈도우 함수                                           │
│ UNION                  │ UNION / UNION ALL                                     │
│ CROSS_PRODUCT          │ CROSS JOIN 또는 조인 키 없는 카테시안 곱               │
└────────────────────────┴──────────────────────────────────────────────────────┘
```

### EXPLAIN ANALYZE 출력 분석

`EXPLAIN ANALYZE`를 사용하면 실제 실행 결과가 추정치와 얼마나 다른지 비교할 수 있다. 아래 예시에서도 동일하게 **아래에서 위로** 읽는다. 먼저 SEQ_SCAN과 FILTER에서 몇 행이 나왔는지 확인하고, HASH_JOIN을 거쳐 HASH_GROUP_BY까지 데이터가 어떻게 줄어드는지 추적하면 된다.

```sql
EXPLAIN ANALYZE
SELECT
    c.region,
    COUNT(DISTINCT o.customer_id) AS unique_customers,
    SUM(o.amount) AS revenue
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.order_date >= '2024-01-01'
GROUP BY c.region;
```

출력 예시:

```
┌─────────────────────────────────────────────────────────────────┐
│        HASH_GROUP_BY                                            │
│        Aggregates: count(DISTINCT customer_id), sum(amount)     │
│        Groups: region                                           │
│─────────────────────────────────────────────────────────────────│
│        Estimated Rows: 20    Actual Rows: 18                    │
│        Execution Time: 12ms                                     │
└──────────────────────┬──────────────────────────────────────────┘
                       │
┌──────────────────────┴──────────────────────────────────────────┐
│        HASH_JOIN (orders.customer_id = customers.id)            │
│─────────────────────────────────────────────────────────────────│
│        Estimated Rows: 500K  Actual Rows: 483,201               │
│        Build Side: customers (작은 쪽이 해시 테이블)             │
│        Probe Side: orders                                       │
│        Execution Time: 340ms                                    │
└──────────┬───────────────────────────┬───────────────────────────┘
           │                           │
  ┌────────┴────────┐         ┌────────┴────────┐
  │   SEQ_SCAN      │         │   FILTER +      │
  │   customers     │         │   SEQ_SCAN      │
  │   Rows: 12,000  │         │   orders        │
  │   Time: 2ms     │         │   Rows: 483,201 │
  └─────────────────┘         │   Time: 580ms   │
                              └─────────────────┘
```

**실측 행 수와 추정 행 수의 괴리**가 크면 통계가 오래된 것이므로 `ANALYZE`를 실행해 통계를 갱신한다. 위 예시에서 HASH_JOIN의 추정치(500K)와 실측치(483,201)가 비슷하다면 통계 상태가 양호하다는 신호다.

### 최적화 힌트: 통계 갱신

```sql
-- 테이블 통계 갱신 후 재실행
ANALYZE orders;
ANALYZE customers;

-- 통계 확인
SELECT
    table_name,
    estimated_size,
    column_count
FROM duckdb_tables()
WHERE table_name IN ('orders', 'customers');
```

### 프로파일링 JSON 출력

```sql
PRAGMA enable_profiling = 'json';
PRAGMA profiling_output = '/tmp/profile.json';

SELECT region, SUM(amount) FROM sales GROUP BY region;

PRAGMA disable_profiling;
```

```python
import json

with open('/tmp/profile.json') as f:
    profile = json.load(f)

# 각 연산자별 소요 시간 추출
def print_timings(node, depth=0):
    name = node.get('name', 'unknown')
    timing = node.get('timing', 0)
    rows = node.get('cardinality', 0)
    print(f"{'  ' * depth}{name}: {timing*1000:.1f}ms, rows={rows:,}")
    for child in node.get('children', []):
        print_timings(child, depth + 1)

print_timings(profile)
```
