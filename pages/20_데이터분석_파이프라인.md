# 21장. 데이터 분석 파이프라인 구축

DuckDB의 진정한 가치는 파이프라인 안에서 드러난다. 로컬 머신 한 대에서 수십 GB의 데이터를 처리하고, S3 위의 Parquet 레이크를 직접 쿼리하며, 결과를 실시간 대시보드로 연결하는 전체 흐름을 DuckDB 하나로 관통할 수 있다. 이 장에서는 실제 운영 가능한 세 가지 파이프라인 패턴을 단계적으로 구축한다.

### 이 장의 전체 흐름

아래 순서도는 이 장에서 다루는 파이프라인 전체 구조를 한눈에 보여준다. 각 섹션은 이 흐름의 한 단계에 해당한다.

```
[원본 데이터]
    │  CSV / JSON / 외부 스트림
    ▼
┌─────────────────────────────┐
│  20.1  로컬 데이터 레이크   │  Bronze → Silver → Gold
│  (메달리온 아키텍처)         │  로컬 파일시스템 + Parquet
└──────────────┬──────────────┘
               │  파이프라인 검증 완료 후
               ▼
┌─────────────────────────────┐
│  20.2  S3 + Parquet 레이크  │  클라우드 확장
│  (서버리스 ETL)              │  httpfs + Hive 파티셔닝
└──────────────┬──────────────┘
               │  Gold 마트 준비 완료 후
               ▼
┌─────────────────────────────┐
│  20.3  대시보드 연동         │  Streamlit / FastAPI
│  (시각화·자동 갱신)          │  Grafana / Superset
└─────────────────────────────┘
```

로컬 레이크에서 구조를 검증하고, S3 레이크로 확장하며, 최종적으로 Gold 마트를 대시보드에 연결하는 흐름이다. 각 섹션은 이전 단계의 결과물을 입력으로 사용하므로 순서대로 읽는 것을 권장한다.

## 20.1 로컬 데이터 레이크 만들기

클라우드 없이도 로컬 파일 시스템 위에 완전한 데이터 레이크를 구성할 수 있다.

### 왜 세 계층으로 나누는가?

**메달리온 아키텍처(Medallion Architecture)**는 데이터를 품질 수준에 따라 Bronze, Silver, Gold 세 계층으로 분리해 관리하는 설계 패턴이다. 계층을 나누는 핵심 이유는 두 가지다. 첫째, 원본 데이터를 손상 없이 보존해 언제든 재처리를 가능하게 한다. 둘째, 각 계층에 명확한 책임을 부여해 파이프라인의 어느 단계에서 문제가 발생했는지 빠르게 파악할 수 있다.

| 계층 | 역할 | 처리 내용 |
|------|------|-----------|
| **Bronze** | 원본 보존 | 수집한 데이터를 가공 없이 Parquet으로 저장. 메타데이터(수집 일시, 출처 파일)만 추가한다. |
| **Silver** | 정제·검증 | 타입 변환, 중복 제거, 유효성 검증을 수행한다. 분석에 바로 쓸 수 있는 신뢰할 수 있는 데이터가 만들어진다. |
| **Gold** | 집계·마트 | 비즈니스 질문에 맞는 집계 테이블을 생성한다. 대시보드나 리포트가 이 계층을 직접 읽는다. |

원본 데이터를 수집(Bronze)하고, 정제(Silver)하고, 집계(Gold)하는 메달리온 아키텍처를 DuckDB와 Parquet으로 구현한다.

### 메달리온 아키텍처 디렉터리 구조

```
/data/lake/
├── bronze/                    # 원본 데이터 (가공 없이 저장)
│   ├── orders/
│   │   ├── year=2024/
│   │   │   ├── month=01/raw_20240101.parquet
│   │   │   └── month=02/raw_20240201.parquet
│   └── customers/
│       └── customers_raw.parquet
│
├── silver/                    # 정제 데이터 (타입 변환, 중복 제거, 유효성 검증)
│   ├── orders/
│   │   └── year=2024/month=01/orders_clean.parquet
│   └── customers/
│       └── customers_clean.parquet
│
└── gold/                      # 집계 데이터 (분석 바로 사용 가능한 마트)
    ├── daily_revenue.parquet
    ├── customer_segments.parquet
    └── product_performance.parquet
```

> **주의:** 파티션 키(`year=`, `month=`)는 나중에 변경하기 어렵다. 쿼리에서 가장 자주 사용하는 필터 기준을 파티션 키로 선택하고, 파티션 수가 지나치게 많아지지 않도록 설계한다. 일(day) 단위 파티션은 데이터가 충분히 클 때만 추가한다.

### Bronze 레이어: 원본 수집

Bronze 레이어의 역할은 단순하다. 데이터를 있는 그대로 받아 Parquet으로 변환하고, 언제 어디서 수집됐는지를 나타내는 메타데이터 컬럼을 추가하는 것이다. 원본을 변형하지 않기 때문에 Silver나 Gold 처리에 문제가 생겨도 언제든 Bronze부터 재처리할 수 있다.

```python
# pipeline/bronze.py - CSV/JSON 원본 데이터를 Parquet으로 수집
import duckdb
from pathlib import Path
from datetime import date, datetime
import logging

logger = logging.getLogger(__name__)

LAKE_ROOT = Path('/data/lake')


def ingest_csv_to_bronze(
    source_path: str,
    table_name: str,
    partition_date: date = None,
):
    """CSV 파일을 Bronze 레이어 Parquet으로 적재"""
    con = duckdb.connect()
    partition_date = partition_date or date.today()

    output_dir = (
        LAKE_ROOT / 'bronze' / table_name
        / f'year={partition_date.year}'
        / f'month={partition_date.month:02d}'
    )
    output_dir.mkdir(parents=True, exist_ok=True)
    output_path = output_dir / f'{table_name}_{partition_date.strftime("%Y%m%d")}.parquet'

    # 원본 그대로 저장 + 수집 메타데이터 컬럼 추가
    row_count = con.execute(f"""
        COPY (
            SELECT
                *,
                '{source_path}' AS _source_file,
                NOW()           AS _ingested_at,
                '{partition_date}' AS _partition_date
            FROM read_csv('{source_path}', auto_detect=true, ignore_errors=true)
        )
        TO '{output_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """).fetchone()

    logger.info(f"Bronze 적재 완료: {output_path}")
    return str(output_path)


def ingest_json_to_bronze(source_path: str, table_name: str, partition_date: date = None):
    """NDJSON 파일을 Bronze 레이어로 적재"""
    con = duckdb.connect()
    partition_date = partition_date or date.today()

    output_dir = (
        LAKE_ROOT / 'bronze' / table_name
        / f'year={partition_date.year}'
        / f'month={partition_date.month:02d}'
    )
    output_dir.mkdir(parents=True, exist_ok=True)
    output_path = output_dir / f'{table_name}_{partition_date.strftime("%Y%m%d")}.parquet'

    con.execute(f"""
        COPY (
            SELECT
                *,
                NOW() AS _ingested_at
            FROM read_json('{source_path}', auto_detect=true)
        )
        TO '{output_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """)
    return str(output_path)
```

### Silver 레이어: 정제 및 검증

Bronze에서 넘어온 데이터를 신뢰할 수 있는 상태로 만드는 단계다. `TRY_CAST`로 타입 변환에 실패한 값을 NULL로 처리하고, `QUALIFY ROW_NUMBER()`로 중복을 제거하며, 비즈니스 규칙에 맞지 않는 레코드를 걸러낸다. 이 단계를 거친 데이터가 Gold 마트의 재료가 된다.

```python
# pipeline/silver.py - Bronze 데이터 정제
import duckdb
from pathlib import Path
from datetime import date
import logging

logger = logging.getLogger(__name__)
LAKE_ROOT = Path('/data/lake')


def transform_orders_to_silver(partition_date: date):
    """주문 데이터 정제: 타입 변환, 중복 제거, 유효성 검증"""
    con = duckdb.connect()

    bronze_path = (
        LAKE_ROOT / 'bronze' / 'orders'
        / f'year={partition_date.year}'
        / f'month={partition_date.month:02d}'
        / '*.parquet'
    )
    output_dir = (
        LAKE_ROOT / 'silver' / 'orders'
        / f'year={partition_date.year}'
        / f'month={partition_date.month:02d}'
    )
    output_dir.mkdir(parents=True, exist_ok=True)
    output_path = output_dir / 'orders_clean.parquet'

    result = con.execute(f"""
        COPY (
            WITH raw AS (
                SELECT *
                FROM read_parquet('{bronze_path}')
            ),
            -- 타입 변환 및 정제
            typed AS (
                SELECT
                    TRY_CAST(order_id   AS BIGINT)    AS order_id,
                    TRY_CAST(customer_id AS BIGINT)   AS customer_id,
                    TRY_CAST(amount     AS DECIMAL(12,2)) AS amount,
                    TRY_CAST(order_date AS DATE)      AS order_date,
                    UPPER(TRIM(status))               AS status,
                    UPPER(TRIM(region))               AS region,
                    _ingested_at,
                    _source_file
                FROM raw
            ),
            -- 유효성 검증 (NULL 핵심 컬럼, 음수 금액 제거)
            valid AS (
                SELECT *
                FROM typed
                WHERE order_id   IS NOT NULL
                  AND customer_id IS NOT NULL
                  AND amount      > 0
                  AND order_date  IS NOT NULL
                  AND status IN ('COMPLETED', 'PENDING', 'CANCELLED', 'REFUNDED')
            ),
            -- 중복 제거 (order_id 기준 최신 레코드 유지)
            deduped AS (
                SELECT *
                FROM valid
                QUALIFY ROW_NUMBER() OVER (
                    PARTITION BY order_id
                    ORDER BY _ingested_at DESC
                ) = 1
            )
            SELECT * FROM deduped
        )
        TO '{output_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """)

    # 품질 리포트
    stats = con.execute(f"""
        SELECT
            COUNT(*) AS total_rows,
            COUNT(DISTINCT order_id) AS unique_orders,
            MIN(order_date) AS earliest_date,
            MAX(order_date) AS latest_date,
            SUM(amount) AS total_amount,
            COUNT(*) FILTER (WHERE status = 'COMPLETED') AS completed_count
        FROM read_parquet('{output_path}')
    """).fetchone()

    logger.info(
        f"Silver 변환 완료: {output_path} | "
        f"rows={stats[0]:,} unique_orders={stats[1]:,} "
        f"total_amount={stats[4]:,.2f}"
    )
    return str(output_path)
```

### Gold 레이어: 분석 마트 생성

Silver의 정제 데이터를 비즈니스 질문 단위로 집계하는 마지막 계층이다. 일별 매출, 고객 세분화처럼 대시보드나 리포트에서 반복적으로 사용되는 집계를 미리 계산해 두어 조회 속도를 극적으로 높인다.

```python
# pipeline/gold.py - Silver 데이터로 분석 마트 생성
import duckdb
from pathlib import Path
import logging

logger = logging.getLogger(__name__)
LAKE_ROOT = Path('/data/lake')


def build_daily_revenue_mart():
    """일별 매출 집계 마트 생성"""
    con = duckdb.connect()

    silver_orders = LAKE_ROOT / 'silver' / 'orders' / '**' / '*.parquet'
    silver_customers = LAKE_ROOT / 'silver' / 'customers' / '*.parquet'
    output_path = LAKE_ROOT / 'gold' / 'daily_revenue.parquet'
    output_path.parent.mkdir(parents=True, exist_ok=True)

    con.execute(f"""
        COPY (
            SELECT
                o.order_date,
                o.region,
                c.segment                            AS customer_segment,
                COUNT(*)                             AS order_count,
                COUNT(DISTINCT o.customer_id)        AS unique_customers,
                SUM(o.amount)                        AS revenue,
                AVG(o.amount)                        AS avg_order_value,
                MEDIAN(o.amount)                     AS median_order_value,
                SUM(SUM(o.amount)) OVER (
                    PARTITION BY o.region
                    ORDER BY o.order_date
                    ROWS UNBOUNDED PRECEDING
                )                                    AS cumulative_revenue,
                NOW()                                AS _built_at
            FROM read_parquet('{silver_orders}', union_by_name=true) o
            LEFT JOIN read_parquet('{silver_customers}') c
                ON o.customer_id = c.customer_id
            WHERE o.status = 'COMPLETED'
            GROUP BY o.order_date, o.region, c.segment
            ORDER BY o.order_date, o.region
        )
        TO '{output_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """)

    logger.info(f"Gold 마트 생성 완료: {output_path}")


def build_customer_segments():
    """고객 세분화 마트 생성 (RFM 분석)"""
    con = duckdb.connect()

    silver_orders = LAKE_ROOT / 'silver' / 'orders' / '**' / '*.parquet'
    output_path = LAKE_ROOT / 'gold' / 'customer_segments.parquet'

    con.execute(f"""
        COPY (
            WITH rfm AS (
                SELECT
                    customer_id,
                    MAX(order_date)                         AS last_order_date,
                    DATE_DIFF('day', MAX(order_date), TODAY()) AS recency_days,
                    COUNT(*)                                AS frequency,
                    SUM(amount)                             AS monetary
                FROM read_parquet('{silver_orders}', union_by_name=true)
                WHERE status = 'COMPLETED'
                GROUP BY customer_id
            ),
            scored AS (
                SELECT
                    *,
                    NTILE(5) OVER (ORDER BY recency_days ASC)  AS r_score,
                    NTILE(5) OVER (ORDER BY frequency DESC)    AS f_score,
                    NTILE(5) OVER (ORDER BY monetary DESC)     AS m_score
                FROM rfm
            )
            SELECT
                customer_id,
                last_order_date,
                recency_days,
                frequency,
                ROUND(monetary, 2) AS monetary,
                r_score, f_score, m_score,
                (r_score + f_score + m_score) AS rfm_total,
                CASE
                    WHEN r_score >= 4 AND f_score >= 4 THEN 'Champions'
                    WHEN r_score >= 3 AND f_score >= 3 THEN 'Loyal Customers'
                    WHEN r_score >= 4 AND f_score <= 2 THEN 'Recent Customers'
                    WHEN r_score <= 2 AND f_score >= 3 THEN 'At Risk'
                    WHEN r_score <= 2 AND f_score <= 2 THEN 'Lost'
                    ELSE 'Potential'
                END AS segment
            FROM scored
        )
        TO '{output_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """)

    logger.info(f"고객 세분화 마트 생성 완료: {output_path}")
```

### 파이프라인 오케스트레이터

Bronze, Silver, Gold 세 단계를 순서대로 실행하는 진입점이다. 각 단계가 완료된 후에야 다음 단계가 시작되므로 중간에 실패해도 원인 파악이 쉽다.

```python
# pipeline/orchestrator.py - 전체 파이프라인 실행
from datetime import date, timedelta
from bronze import ingest_csv_to_bronze
from silver import transform_orders_to_silver
from gold import build_daily_revenue_mart, build_customer_segments
import logging

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s %(levelname)s %(message)s'
)


def run_daily_pipeline(target_date: date = None):
    target_date = target_date or date.today() - timedelta(days=1)
    print(f"\n{'='*50}")
    print(f" 파이프라인 실행: {target_date}")
    print(f"{'='*50}")

    # 1. Bronze: 원본 수집
    print("\n[1/3] Bronze 레이어 수집 중...")
    ingest_csv_to_bronze(
        f'/raw/orders_{target_date.strftime("%Y%m%d")}.csv',
        'orders',
        target_date
    )

    # 2. Silver: 정제
    print("[2/3] Silver 레이어 정제 중...")
    transform_orders_to_silver(target_date)

    # 3. Gold: 집계 마트 갱신
    print("[3/3] Gold 레이어 마트 갱신 중...")
    build_daily_revenue_mart()
    build_customer_segments()

    print(f"\n파이프라인 완료: {target_date}\n")


if __name__ == '__main__':
    import sys
    if len(sys.argv) > 1:
        run_daily_pipeline(date.fromisoformat(sys.argv[1]))
    else:
        run_daily_pipeline()
```

## 20.2 S3 + Parquet + DuckDB 분석 환경

로컬 레이크에서 구조와 흐름을 검증했다면, 이제 동일한 Bronze → Silver → Gold 패턴을 클라우드로 확장할 차례다. DuckDB의 `httpfs` 확장을 사용하면 S3 파일을 로컬에 다운로드하지 않고 필요한 Row Group만 선택적으로 읽어올 수 있어 비용과 처리 시간을 동시에 줄인다.

클라우드 스토리지 위에 데이터를 쌓고 DuckDB로 직접 쿼리하는 패턴은 별도의 데이터 웨어하우스 없이도 페타바이트급 분석 환경을 구성할 수 있다.

### 분석 환경 아키텍처

```
┌─────────────────────────────────────────────────────────────────┐
│                      S3 데이터 레이크                            │
│                                                                  │
│  s3://my-lake/                                                   │
│  ├── bronze/   (원본 CSV, JSON → Parquet 변환)                   │
│  ├── silver/   (정제 Parquet, Hive 파티셔닝)                     │
│  └── gold/     (집계 마트 Parquet)                               │
└──────────────────────────┬──────────────────────────────────────┘
                           │  httpfs 확장
                           │  S3 Range Request
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                    DuckDB (로컬 / 컨테이너)                       │
│                                                                  │
│  - Zone Map으로 불필요한 Row Group 스킵                          │
│  - Hive 파티션 프루닝으로 불필요한 파일 스킵                      │
│  - Projection Pushdown으로 필요한 컬럼만 전송                    │
│  - 멀티스레드 병렬 다운로드                                       │
└─────────────────────────────────────────────────────────────────┘
```

### S3 연결 설정

```python
# config/s3_setup.py
import duckdb
import os


def create_s3_connection(profile: str = 'prod') -> duckdb.DuckDBPyConnection:
    con = duckdb.connect(config={
        'threads': os.cpu_count(),
        'memory_limit': '16GB',
        'temp_directory': '/tmp/duckdb_spill',
    })

    con.execute("INSTALL httpfs; LOAD httpfs;")

    if os.environ.get('AWS_EXECUTION_ENV'):
        # EC2/ECS: IAM 역할 자동 감지
        con.execute("""
            CREATE OR REPLACE SECRET s3 (
                TYPE S3,
                PROVIDER CREDENTIAL_CHAIN
            )
        """)
    else:
        # 로컬 개발: 환경 변수 또는 ~/.aws/credentials
        con.execute(f"""
            CREATE OR REPLACE SECRET s3 (
                TYPE S3,
                PROVIDER CREDENTIAL_CHAIN,
                CHAIN 'env;config'
            )
        """)

    # HTTP 재시도 및 캐시 설정
    con.execute("SET http_retries = 3")
    con.execute("SET http_retry_wait_ms = 1000")

    return con
```

### S3 기반 ETL 파이프라인

```python
# pipeline/s3_etl.py
import duckdb
from config.s3_setup import create_s3_connection
from datetime import date, timedelta

LAKE_BUCKET = 's3://my-data-lake'


def s3_bronze_to_silver(target_date: date):
    """S3 Bronze → Silver 변환 (서버리스 ETL)"""
    con = create_s3_connection()

    bronze_path = (
        f"{LAKE_BUCKET}/bronze/orders"
        f"/year={target_date.year}"
        f"/month={target_date.month:02d}"
        f"/day={target_date.day:02d}"
        f"/*.parquet"
    )
    silver_path = (
        f"{LAKE_BUCKET}/silver/orders"
        f"/year={target_date.year}"
        f"/month={target_date.month:02d}"
        f"/day={target_date.day:02d}"
        f"/orders_clean.parquet"
    )

    # S3 파일 존재 확인
    files = con.execute(f"SELECT * FROM glob('{bronze_path}')").fetchall()
    if not files:
        print(f"처리할 파일 없음: {bronze_path}")
        return

    print(f"처리 대상: {len(files)}개 파일")

    # S3 → S3 직접 변환 (로컬 다운로드 없이)
    con.execute(f"""
        COPY (
            WITH cleaned AS (
                SELECT
                    TRY_CAST(order_id    AS BIGINT)       AS order_id,
                    TRY_CAST(customer_id AS BIGINT)       AS customer_id,
                    TRY_CAST(amount      AS DECIMAL(12,2)) AS amount,
                    TRY_CAST(order_date  AS DATE)         AS order_date,
                    UPPER(TRIM(status))                   AS status,
                    UPPER(TRIM(region))                   AS region
                FROM read_parquet('{bronze_path}')
                WHERE order_id IS NOT NULL
                  AND amount > 0
            )
            SELECT * FROM cleaned
            QUALIFY ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY order_date DESC) = 1
        )
        TO '{silver_path}' (FORMAT PARQUET, CODEC 'ZSTD')
    """)
    print(f"Silver 저장 완료: {silver_path}")


def s3_adhoc_analysis(start_date: date, end_date: date):
    """S3 Silver 데이터 임시 분석"""
    con = create_s3_connection()

    result = con.execute(f"""
        SELECT
            DATE_TRUNC('week', order_date) AS week,
            region,
            COUNT(*)                        AS orders,
            COUNT(DISTINCT customer_id)     AS unique_customers,
            ROUND(SUM(amount), 2)           AS revenue,
            ROUND(AVG(amount), 2)           AS avg_order
        FROM read_parquet(
            '{LAKE_BUCKET}/silver/orders/**/*.parquet',
            hive_partitioning = true
        )
        WHERE order_date BETWEEN '{start_date}' AND '{end_date}'
          AND status = 'COMPLETED'
        GROUP BY 1, 2
        ORDER BY 1, revenue DESC
    """).df()

    return result
```

### S3 파티션 관리와 소형 파일 병합

> **주의:** S3 환경에서 파이프라인을 오래 운영하면 파티션마다 수백 개의 작은 Parquet 파일이 쌓이는 "소형 파일 문제(small file problem)"가 발생한다. 파일 수가 많아지면 S3 LIST 요청 비용이 증가하고 DuckDB의 파티션 프루닝 효율도 떨어진다. 월 1회 이상 `compact_silver_partition`을 실행해 파티션을 정리하는 것을 권장한다.

```python
# maintenance/compact.py - 소형 Parquet 파일 병합
import duckdb
from config.s3_setup import create_s3_connection

LAKE_BUCKET = 's3://my-data-lake'


def compact_silver_partition(year: int, month: int):
    """월별 Silver 파티션의 소형 파일을 하나로 병합"""
    con = create_s3_connection()

    source_path = f"{LAKE_BUCKET}/silver/orders/year={year}/month={month:02d}/**/*.parquet"
    target_path = f"{LAKE_BUCKET}/silver/orders_compacted/year={year}/month={month:02d}/data.parquet"

    # 파일 크기 확인
    files = con.execute(f"""
        SELECT
            filename,
            file_size_bytes / 1e6 AS size_mb
        FROM parquet_file_metadata('{source_path}')
        ORDER BY size_mb
    """).fetchdf()

    print(f"병합 대상: {len(files)}개 파일, "
          f"총 {files['size_mb'].sum():.1f}MB")

    # 모든 파일을 하나로 병합
    con.execute(f"""
        COPY (
            SELECT * FROM read_parquet('{source_path}', union_by_name=true)
            ORDER BY order_date, customer_id
        )
        TO '{target_path}' (
            FORMAT PARQUET,
            CODEC 'ZSTD',
            ROW_GROUP_SIZE 100000
        )
    """)
    print(f"병합 완료: {target_path}")


def list_small_partitions(threshold_mb: float = 10.0):
    """소형 파일이 많은 파티션 목록 조회"""
    con = create_s3_connection()

    return con.execute(f"""
        SELECT
            regexp_extract(filename, 'year=(\d+)/month=(\d+)', 0) AS partition,
            COUNT(*) AS file_count,
            ROUND(SUM(file_size_bytes) / 1e6, 1) AS total_mb,
            ROUND(AVG(file_size_bytes) / 1e6, 1) AS avg_file_mb
        FROM parquet_file_metadata(
            '{LAKE_BUCKET}/silver/orders/**/*.parquet'
        )
        GROUP BY 1
        HAVING avg_file_mb < ?
        ORDER BY avg_file_mb
    """, [threshold_mb]).df()
```

### 비용 최적화: S3 요청 수 줄이기

```python
def cost_optimized_query(con, date_filter: str):
    """S3 비용을 최소화하는 쿼리 패턴"""

    # 나쁜 예: 전체 파일 스캔 후 Python 필터
    # → 모든 파일을 S3에서 다운로드
    bad = con.execute(f"""
        SELECT * FROM read_parquet('{LAKE_BUCKET}/silver/orders/**/*.parquet')
    """).df()
    result = bad[bad['order_date'] >= date_filter]   # Python 필터 (이미 전부 다운로드됨)

    # 좋은 예: Hive 파티셔닝 + WHERE로 S3 요청 최소화
    # → 해당 날짜 파티션 파일만 다운로드
    good = con.execute(f"""
        SELECT *
        FROM read_parquet(
            '{LAKE_BUCKET}/silver/orders/**/*.parquet',
            hive_partitioning = true
        )
        WHERE order_date >= '{date_filter}'
    """).df()

    return good
```

# dashboard/grafana_api.py - Grafana JSON 데이터소스 형식 API
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List, Optional
import duckdb
from datetime import datetime
from pathlib import Path

app = FastAPI()
LAKE_ROOT = Path('/data/lake')
_con = duckdb.connect(config={'threads': 4, 'memory_limit': '4GB'})


class TimeRange(BaseModel):
    from_: str
    to: str

    class Config:
        fields = {'from_': 'from'}


class QueryRequest(BaseModel):
    targets: List[dict]
    range: dict
    intervalMs: int = 60000


@app.get('/')
def health():
    return {'status': 'ok'}


@app.post('/query')
def query(req: QueryRequest):
    """Grafana Simple JSON 형식 응답"""
    start = req.range['from'][:10]
    end   = req.range['to'][:10]
    results = []

    for target in req.targets:
        metric = target.get('target', 'revenue')

        df = _con.execute(f"""
            SELECT
                order_date,
                SUM({metric}) AS value
            FROM read_parquet('{LAKE_ROOT}/gold/daily_revenue.parquet')
            WHERE order_date BETWEEN '{start}' AND '{end}'
            GROUP BY order_date
            ORDER BY order_date
        """).df()

        # Grafana timeseries 형식: [[value, timestamp_ms], ...]
        datapoints = [
            [row['value'], int(row['order_date'].timestamp() * 1000)]
            for _, row in df.iterrows()
        ]
        results.append({'target': metric, 'datapoints': datapoints})

    return results


@app.post('/search')
def search():
    """사용 가능한 메트릭 목록 반환"""
    return ['revenue', 'orders', 'unique_customers', 'avg_order_value']

## 정리

| 카테고리 | 핵심 패턴 | 주요 포인트 |
|----------|-----------|------------|
| Bronze 수집 | `read_csv` + 메타데이터 컬럼 추가 | `_source_file`, `_ingested_at` 트레이서빌리티 확보 |
| Silver 정제 | `TRY_CAST` + `QUALIFY ROW_NUMBER()` | 타입 변환 실패 시 NULL, 중복은 최신 레코드 유지 |
| Gold 마트 | 윈도우 함수 + LEFT JOIN | 누적 집계, RFM 세분화 단일 쿼리로 처리 |
| S3 연결 | `CREDENTIAL_CHAIN` + `hive_partitioning` | IAM 자동 감지, 파티션 프루닝으로 비용 절감 |
| S3 병합 | `parquet_file_metadata` + COPY | 소형 파일 탐지 후 배치 병합 |
| Streamlit | `@st.cache_resource` + `@st.cache_data(ttl=)` | 연결은 세션 당 1회, 쿼리는 TTL 캐싱 |
| FastAPI | lifespan 연결 관리 + Plotly HTML | 단일 연결로 REST API + HTML 대시보드 |
| Superset | `duckdb-engine` SQLAlchemy 드라이버 | 연결 문자열 `duckdb:////path/to/db` |
| Grafana | Simple JSON 플러그인 형식 | `[[value, timestamp_ms]]` 배열 형식 준수 |
| 자동 갱신 | crontab + 캐시 무효화 API | ETL 완료 직후 대시보드 캐시 초기화 |

로컬 데이터 레이크에서 S3 기반 클라우드 레이크까지, DuckDB는 수집·변환·집계·시각화의 전 과정에서 단일 쿼리 엔진으로 동작한다. 별도의 클러스터나 복잡한 인프라 없이 노트북 한 대로 시작해 클라우드로 자연스럽게 확장할 수 있다는 것이 DuckDB 파이프라인의 가장 큰 강점이다.

## 마치며

이 장에서는 DuckDB를 중심으로 실제 운영 가능한 데이터 파이프라인을 단계적으로 구축했다. 메달리온 아키텍처의 세 계층(Bronze, Silver, Gold)이 각각 원본 보존, 품질 정제, 비즈니스 집계라는 명확한 책임을 가짐으로써 파이프라인의 어느 단계에서도 문제를 빠르게 격리하고 재처리할 수 있다는 점을 확인했다. 로컬 파일시스템에서 검증한 동일한 패턴이 S3로 그대로 확장되고, Gold 마트에 쌓인 집계 결과가 Streamlit, FastAPI, Grafana, Superset 등 다양한 시각화 도구로 자연스럽게 이어진다는 것도 살펴봤다. 파티션 설계와 소형 파일 병합은 초기부터 고려해야 나중에 드는 비용과 운영 부담을 크게 줄일 수 있다.

**이 장의 핵심 포인트**

- Bronze는 원본을 절대 변형하지 않는다. 재처리 가능성이 파이프라인 안정성의 핵심이다.
- S3 비용 최적화는 쿼리 작성 방식에서 결정된다. Hive 파티셔닝과 `WHERE` 조건을 함께 써야 파일 스캔을 줄일 수 있다.
- 대시보드 연결은 Gold 마트를 통해 이루어지며, 연결 방식과 무관하게 캐시 전략이 응답 속도를 좌우한다.

23장에서는 DuckDB를 프로덕션 환경에 배포하고 운영하는 방법, 즉 모니터링, 접근 제어, 백업 전략을 살펴본다.
