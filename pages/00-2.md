# 00-2. 왜 DuckDB인가: SQLite vs PostgreSQL vs DuckDB

"그냥 SQLite나 PostgreSQL 쓰면 되는 거 아닌가?" 하는 질문이 당연히 나온다. 각 데이터베이스가 어떤 문제를 풀기 위해 만들어졌는지를 이해하면 답이 분명해진다.

### SQLite: 트랜잭션의 강자, 분석의 약자

SQLite는 임베디드 데이터베이스의 원조다. 스마트폰 앱, 웹 브라우저, IoT 기기 등 수십억 개의 장치에서 조용히 동작하고 있다. 단건 읽기·쓰기, 소규모 데이터 영속화—이런 OLTP 시나리오에서 SQLite는 탁월하다.

그러나 수천만 행을 집계하거나, 여러 테이블을 복잡하게 조인하거나, 윈도우 함수로 순위를 매기는 분석 쿼리를 SQLite에 던지면 고통스러울 만큼 느려진다. SQLite는 **행(row) 기반 스토리지**다. 분석 쿼리가 특정 컬럼만 읽어도 행 전체를 읽어야 한다. 벡터 실행도, 병렬 처리도 없다.

### PostgreSQL: 강력한 서버 DB, 하지만 서버가 필요하다

PostgreSQL은 강력한 오픈소스 관계형 데이터베이스다. 복잡한 SQL, 다양한 확장, 강력한 트랜잭션 보장—엔터프라이즈 OLTP 워크로드에 오랫동안 검증된 선택지다. 분석 기능도 갖추고 있으며, 충분한 하드웨어와 튜닝이 뒷받침되면 분석 쿼리도 빠르게 처리할 수 있다.

문제는 **오버헤드**다. PostgreSQL은 서버 프로세스를 띄워야 하고, 연결 설정이 필요하며, 데이터를 가져오려면 네트워크 통신을 거쳐야 한다. Pandas DataFrame에 있는 데이터를 분석하려면 DB에 먼저 적재해야 한다. 노트북에서 빠르게 탐색적 분석을 하기엔 준비 비용이 너무 크다.

### DuckDB: 분석 전용, 임베디드, 고성능

| 항목 | SQLite | PostgreSQL | DuckDB |
|---|---|---|---|
| 아키텍처 | 임베디드 | 클라이언트-서버 | 임베디드 |
| 최적화 대상 | OLTP | OLTP (+ OLAP 가능) | OLAP |
| 스토리지 방식 | 행(Row) 기반 | 행(Row) 기반 | 컬럼(Column) 기반 |
| 병렬 처리 | 없음 | 제한적 | 멀티코어 기본 지원 |
| 외부 파일 쿼리 | 불가 | 불가 (FDW 필요) | 기본 지원 |
| 서버 설치 | 불필요 | 필요 | 불필요 |
| 대용량 분석 쿼리 | 느림 | 보통 | 빠름 |
| 라이선스 | Public Domain | PostgreSQL License | MIT |

DuckDB는 SQLite의 **간편함**과 PostgreSQL의 **SQL 완성도**를 합치고, 거기에 분석 전용 엔진의 **속도**를 더한 형태라고 볼 수 있다. "모든 데이터베이스를 대체한다"는 주장이 아니다. 분석이라는 특정 문제를 풀기 위해 아키텍처 전체를 최적화한 도구다.

> **주의:** DuckDB는 동시에 하나의 프로세스만 데이터베이스 파일에 쓸 수 있다. 여러 프로세스가 동시에 같은 파일을 수정해야 하는 상황이라면 PostgreSQL 같은 클라이언트-서버 DB가 더 적합하다. DuckDB의 강점은 단일 프로세스에서 대용량 읽기·분석이 필요한 상황에 있다.

특히 다음과 같은 상황에서 DuckDB는 탁월한 선택이 된다.

- 서버 없이 로컬에서 대용량 데이터를 빠르게 분석하고 싶을 때
- Pandas/Polars 같은 Python 데이터 도구와 함께 SQL을 쓰고 싶을 때
- CSV, Parquet, JSON 파일을 DB에 적재하지 않고 바로 쿼리하고 싶을 때
- S3나 HTTP 원격 파일에 직접 SQL을 던지고 싶을 때
- 데이터 파이프라인 중간 단계에서 무거운 집계와 변환이 필요할 때
