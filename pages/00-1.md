# 00-1. DuckDB란 무엇인가

데이터 분석의 세계는 빠르게 변하고 있다. 수십 기가바이트, 수백 기가바이트에 이르는 데이터를 노트북 한 대에서 분석해야 하는 일이 점점 더 일상이 되었다. 그런데 막상 도구를 고르려 하면 선택지가 너무 많고, 저마다 장단점이 명확하다. 서버를 띄우기엔 과하고, 그렇다고 단순한 파일 DB로는 부족한 그 어딘가—DuckDB는 바로 그 자리를 정확히 파고든 데이터베이스다.

DuckDB는 2019년 네덜란드 암스테르담 대학교의 Mark Raasveldt와 Hannes Mühleisen이 논문과 함께 공개한 **임베디드 분석 데이터베이스**다. 현재는 DuckDB 재단(DuckDB Foundation)의 관리 아래 MIT 라이선스로 배포되는 완전한 오픈소스 프로젝트이며, Python·R·Go·Rust·Java·Node.js·WebAssembly 등 거의 모든 언어에서 사용할 수 있다.

한 문장으로 정의하자면 이렇다.

> **DuckDB는 별도의 서버 없이 프로세스 안에 내장되어 동작하는, OLAP(분석 처리) 전용 SQL 데이터베이스다.**

여기서 핵심 키워드는 두 가지다. 각 키워드가 실제 개발 경험에서 어떤 의미를 갖는지 잠깐 짚고 넘어가자.

**임베디드(Embedded):** DuckDB는 설치할 서버가 없다. 라이브러리처럼 애플리케이션 안에 포함되어 동작한다. `pip install duckdb` 한 줄이면 Python 환경에서 바로 쓸 수 있고, CLI 바이너리 하나만 받아도 즉시 쿼리를 실행할 수 있다. 덕분에 환경 구성에 드는 시간을 거의 0으로 줄이고 분석 자체에 집중할 수 있다.

**분석(OLAP) 최적화:** DuckDB의 쿼리 엔진은 **컬럼 기반 벡터 실행(columnar-vectorized execution)** 방식으로 동작한다. 수천만 행을 집계하고 조인하는 복잡한 분석 쿼리를 빠르게 처리하도록 설계되었다. 단건 INSERT/UPDATE가 빈번한 트랜잭션 처리(OLTP)보다, 대량 데이터를 훑어 인사이트를 뽑아내는 분석 워크로드에서 빛을 발한다.

> **팁:** OLTP와 OLAP의 차이가 낯설다면 이렇게 기억하자. OLTP는 "한 번에 한 건씩 빠르게 읽고 쓰는 것"(은행 거래, 쇼핑몰 주문), OLAP는 "수백만 건을 한꺼번에 집계해 인사이트를 뽑는 것"(월별 매출 분석, 사용자 행동 패턴)이다. DuckDB는 후자를 위해 설계되었다.

DuckDB의 또 다른 강점은 외부 데이터를 직접 쿼리할 수 있다는 점이다. CSV, JSON, Parquet 파일을 미리 로딩할 필요 없이 `SELECT * FROM 'data.parquet'` 한 줄로 바로 읽을 수 있다. S3나 HTTPS 원격 파일도 마찬가지다. Pandas DataFrame이나 Polars DataFrame도 복사 없이 직접 쿼리 대상이 된다.
