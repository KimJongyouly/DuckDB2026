# 07-1. CSV 읽기

DuckDB가 CSV를 다루는 핵심 철학은 "파일을 로드하지 말고, 바로 쿼리하라"이다. 별도의 import 단계 없이 파일 경로를 테이블처럼 FROM 절에 넣으면 DuckDB가 구분자, 헤더, 컬럼 타입을 스스로 파악하고 즉시 SQL 결과를 돌려준다.

### 가장 간단한 읽기

```sql
-- 파일을 직접 쿼리 (테이블처럼)
SELECT * FROM 'sales.csv';
SELECT * FROM 'data/employees.csv' LIMIT 10;

-- read_csv_auto: 모든 것을 자동으로 감지
SELECT * FROM read_csv_auto('sales.csv');

-- read_csv: 옵션을 명시적으로 지정
SELECT * FROM read_csv('sales.csv', header=true, delim=',');
```

> **팁:** `FROM 'file.csv'`와 `read_csv_auto()`는 동작이 거의 같다. 빠른 탐색에는 파일 경로 직접 입력 방식을, 옵션을 추가해야 할 때는 `read_csv()`를 사용한다.

```
DuckDB CSV 자동 감지 항목
────────────────────────────────────────
✓ 구분자   , / ; / \t / | 등
✓ 헤더 행  존재 여부 자동 판단
✓ 인용 문자 " / ' 자동 감지
✓ 인코딩   UTF-8, Latin-1 등
✓ 날짜 형식 YYYY-MM-DD, MM/DD/YYYY 등
✓ 숫자 형식 천 단위 구분자, 소수점 기호
✓ NULL 표현 빈 문자열, "NULL", "NA", "N/A"
```

### read_csv 주요 옵션

자동 감지가 실패하거나, 파일 구조를 정확히 알고 있을 때는 옵션을 명시적으로 지정하면 DuckDB가 지정된 규칙대로만 파싱하여 처리 속도와 정확도를 높인다. 아래는 상황별로 자주 쓰이는 옵션들이다.

```sql
SELECT * FROM read_csv(
    'sales.csv',

    -- 파일 구조: 구분자·헤더·인용 방식이 표준과 다를 때 지정
    header       = true,          -- 첫 행을 헤더로
    delim        = ',',           -- 구분자 (기본값: ',')
    quote        = '"',           -- 인용 문자
    escape       = '"',           -- 이스케이프 문자 (기본: quote와 동일)
    newline      = '\n',          -- 줄 바꿈 문자
    skip         = 2,             -- 앞 N행 건너뛰기 (메타정보 행이 있을 때)
    comment      = '#',           -- 이 문자로 시작하는 행 무시

    -- 타입 관련: 날짜·숫자 형식이 지역별로 다를 때 유용
    all_varchar  = false,         -- true면 모든 열을 VARCHAR로 읽기
    dateformat   = '%d/%m/%Y',    -- 날짜 파싱 형식 지정 (유럽식 등)
    timestampformat = '%Y-%m-%d %H:%M:%S',

    -- NULL 처리: 소스 시스템마다 NULL 표현이 다를 때 일괄 지정
    nullstr      = ['', 'NULL', 'NA', 'N/A', '-'],

    -- 인코딩: UTF-8이 아닌 파일(EUC-KR 등)을 처리할 때
    encoding     = 'utf-8',       -- 'latin1', 'utf-16' 등

    -- 열 이름 충돌: 헤더에 공백·특수문자가 섞인 파일에 사용
    normalize_names = true        -- 특수문자 → 밑줄 변환
);
```

> **팁:** 옵션을 너무 많이 지정하면 오히려 자동 감지를 방해할 수 있다. 먼저 `read_csv_auto()`로 감지 결과를 확인한 뒤, 실제로 틀린 부분만 옵션으로 보정하는 방식이 효과적이다.

### 타입 자동 감지 동작 원리

자동 감지 결과가 예상과 다를 때는 DuckDB 내부에서 어떤 순서로 타입을 결정하는지 이해하면 원인을 쉽게 찾을 수 있다.

```
CSV 자동 타입 감지 흐름
───────────────────────────────────────────────────
1. 샘플 행 읽기 (기본 20,480행)
2. 각 열의 값을 분석
3. 후보 타입 목록에서 가장 좁은 타입 선택
      BIGINT → INTEGER → SMALLINT
      DOUBLE → FLOAT
      DATE → TIMESTAMP
      BOOLEAN
      VARCHAR (폴백)
4. 전체 파일 읽으며 검증
5. 실패 시 VARCHAR로 강등
```

> **주의:** 샘플 구간(기본 20,480행)에 모든 타입 변형이 등장하지 않으면 감지가 틀릴 수 있다. 예를 들어 앞 2만 행이 모두 정수인데 나중에 소수가 나오면 DOUBLE로 강등된다. 이런 경우 `sniff_csv()`로 먼저 확인하거나 타입을 직접 지정한다.

```sql
-- 감지된 타입 미리 확인
DESCRIBE SELECT * FROM 'sales.csv';

-- 또는 스니퍼로 확인
SELECT * FROM sniff_csv('sales.csv');
-- column_id | column_name | column_type | ...
```

### 타입 명시 지정

자동 감지 결과가 만족스럽지 않을 때, 또는 금융 데이터처럼 소수점 자리수를 정확히 제어해야 할 때는 열별로 타입을 직접 지정한다. DuckDB는 지정된 타입으로만 파싱을 시도하므로 불필요한 타입 추론 과정을 건너뛰어 처리 속도도 빨라진다.

```sql
-- 열별 타입 지정
SELECT * FROM read_csv(
    'orders.csv',
    columns = {
        'order_id':   'INTEGER',
        'order_date': 'DATE',
        'amount':     'DECIMAL(10,2)',
        'status':     'VARCHAR',
        'customer_id':'BIGINT'
    }
);

-- 일부만 지정 (나머지는 자동 감지)
SELECT * FROM read_csv(
    'orders.csv',
    header = true,
    types  = {'amount': 'DECIMAL(10,2)', 'order_date': 'DATE'}
);

-- 전체를 VARCHAR로 읽은 뒤 수동 변환 (최대 안전)
SELECT
    order_id::INTEGER,
    TRY_CAST(order_date AS DATE) AS order_date,
    TRY_CAST(amount AS DECIMAL(10,2)) AS amount
FROM read_csv('orders.csv', all_varchar=true);
```

### 여러 파일 한 번에 읽기

월별·지역별로 쪼개진 CSV 파일들을 하나씩 로드한 뒤 UNION ALL로 합치는 작업은 번거롭다. DuckDB는 와일드카드와 파일 배열을 지원하여 여러 파일을 단일 쿼리로 한꺼번에 처리하고, 어느 파일에서 온 행인지 추적하는 기능도 제공한다.

```sql
-- 와일드카드
SELECT * FROM 'data/sales_*.csv';
SELECT * FROM read_csv_auto('logs/2024_*.csv');

-- 파일 목록 배열
SELECT * FROM read_csv(['jan.csv', 'feb.csv', 'mar.csv']);

-- 어느 파일에서 왔는지 추적
SELECT filename, * FROM read_csv('data/*.csv', filename=true);
-- filename         | id | name | ...
-- ─────────────────┼────┼──────┼───
-- data/jan.csv     |  1 | ...
-- data/feb.csv     | 11 | ...
```

> **팁:** 와일드카드로 여러 파일을 읽을 때 컬럼 수나 타입이 파일마다 다르면 오류가 발생한다. 이 경우 `all_varchar=true`로 일단 모두 읽은 뒤 정제하는 방법이 안전하다.

### 구분자가 다른 형식

```sql
-- TSV (탭 구분)
SELECT * FROM read_csv('data.tsv', delim='\t');
SELECT * FROM 'data.tsv';   -- .tsv 확장자는 자동 감지

-- 세미콜론 구분 (유럽식 CSV)
SELECT * FROM read_csv('euro_data.csv', delim=';');

-- 파이프 구분
SELECT * FROM read_csv('fixed.csv', delim='|');

-- 고정 너비 (Fixed-Width) — 공백으로 파싱 후 처리
SELECT
    TRIM(SUBSTR(line, 1,  10)) AS id,
    TRIM(SUBSTR(line, 11, 30)) AS name,
    TRIM(SUBSTR(line, 41, 12)) AS amount
FROM read_csv('fixed.txt', columns={'line': 'VARCHAR'}, header=false);
```

### 인코딩 처리

DuckDB의 기본 인코딩은 UTF-8이다. 한국어가 포함된 오래된 파일이나 EUC-KR로 저장된 레거시 데이터를 읽을 때는 인코딩을 명시하거나 사전에 변환하는 과정이 필요하다.

```sql
-- EUC-KR 한국어 파일 (iconv로 변환 후 읽기)
-- 방법 1: 변환 스크립트 실행 후 읽기
-- iconv -f EUC-KR -t UTF-8 input.csv > input_utf8.csv

-- 방법 2: DuckDB에서 직접 인코딩 지정
SELECT * FROM read_csv('korean.csv', encoding='latin1');

-- BOM 포함 UTF-8 (Excel 저장 파일)
SELECT * FROM read_csv('excel_export.csv');
-- DuckDB가 BOM을 자동으로 처리
```

> **주의:** DuckDB는 EUC-KR을 직접 지원하지 않는다. `encoding='latin1'`으로 임시 우회가 가능하지만 한글이 깨질 수 있다. 가장 확실한 방법은 `iconv`로 UTF-8 변환 후 읽는 것이다.

### CSV를 테이블로 적재

```sql
-- CTAS: CSV에서 직접 테이블 생성
CREATE TABLE sales AS
SELECT * FROM read_csv_auto('sales.csv');

-- INSERT INTO: 기존 테이블에 추가
INSERT INTO sales
SELECT * FROM read_csv_auto('new_sales.csv');

-- COPY FROM: 고성능 적재
COPY sales FROM 'sales.csv' (HEADER, DELIMITER ',');
COPY sales FROM 'data/*.csv' (HEADER, FORMAT CSV);
```
