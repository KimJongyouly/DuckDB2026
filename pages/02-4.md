# 02-4. 디스크 스필링과 대용량 처리

DuckDB를 처음 쓰면 "RAM이 16GB인데 100GB 파일을 분석할 수 있을까?"라는 의문이 생긴다. 인메모리 모드이건 영구 모드이건, DuckDB는 **처리 중인 데이터가 가용 메모리를 초과하면 자동으로 임시 파일에 데이터를 흘려 보낸다.** 이것이 **디스크 스필링(Spilling to Disk)**, 또는 **out-of-core processing**이다.

이 기능 덕분에 DuckDB는 RAM 용량보다 훨씬 큰 데이터셋도 처리할 수 있다.

```
쿼리 처리 흐름 (메모리 초과 시)

  데이터 → 메모리 버퍼 → [초과 발생] → 임시 디스크 파일
                                              ↓
                                         병합 & 처리
                                              ↓
                                          최종 결과
```

---

### 임시 파일 위치

| 모드 | 기본 임시 디렉토리 |
|---|---|
| 영구 모드 | `<database_name>.tmp/` |
| 인메모리 모드 | `.tmp/` (현재 디렉토리) |

임시 디렉토리를 직접 지정하려면:

```sql
-- SQL로 설정
SET temp_directory = '/fast_ssd/duckdb_temp';
```

```python
# Python 연결 시 설정
con = duckdb.connect(config={
    "temp_directory": "/fast_ssd/duckdb_temp"
})
```

I/O가 빠른 SSD 경로를 지정하면 스필링 성능이 크게 향상된다.

> **팁:** HDD 대신 NVMe SSD에 임시 경로를 지정하면 스필링이 발생해도 쿼리 속도 저하를 최소화할 수 있다. 클라우드 환경이라면 인스턴스 로컬 스토리지(ephemeral disk)를 임시 경로로 활용하는 것이 좋다.

스필링 동작을 더 세밀하게 제어하고 싶다면 메모리와 스레드 관련 설정값을 조정할 수 있다.

---

### 메모리와 스필링 관련 주요 설정

```sql
-- 메모리 상한 설정 (기본값: 시스템 RAM의 80%)
SET memory_limit = '8GB';

-- 임시 파일 최대 용량 (기본값: 사용 가능 디스크의 90%)
SET max_temp_directory_size = '100GB';

-- 사용할 CPU 스레드 수 (기본값: 전체 코어 수)
SET threads = 8;
```

```python
con = duckdb.connect(config={
    "memory_limit": "8GB",
    "threads": 8,
    "temp_directory": "/tmp/duckdb"
})
```

---

### 대용량 처리 시나리오 예

16GB RAM 환경에서 100GB Parquet 파일을 집계하는 경우:

```python
import duckdb

con = duckdb.connect(config={
    "memory_limit": "12GB",          # RAM의 75% 할당
    "temp_directory": "/data/tmp",   # 여유 있는 디스크 경로
    "threads": 16                    # 멀티코어 최대 활용
})

# 100GB 파일도 쿼리 가능 — 필요 시 /data/tmp 에 스필링
result = con.sql("""
    SELECT
        year,
        category,
        SUM(amount) AS total
    FROM 'large_dataset.parquet'
    GROUP BY year, category
""").df()
```

DuckDB는 스필링 여부를 자동으로 결정하고 처리한다. 별도의 코드 변경 없이 동일한 쿼리가 메모리 내에서 처리되기도 하고, 디스크를 활용해 처리되기도 한다. 다만 스필링이 발생하면 I/O 때문에 쿼리 속도가 느려질 수 있으므로, 가능하면 메모리 내에서 처리되도록 설정값을 조율하는 것이 좋다.

> **주의:** `memory_limit`을 시스템 RAM 전체로 설정하면 OS와 다른 프로세스가 사용할 메모리가 부족해져 시스템 전체가 느려질 수 있다. 시스템 RAM의 70~80% 수준으로 설정하는 것이 일반적으로 안전하다.

연결 방식과 대용량 처리를 이해했다면, 이제 여러 프로세스나 스레드가 동시에 DuckDB에 접근할 때 어떤 규칙이 적용되는지 살펴볼 차례다.
