# 03-3. WHERE, GROUP BY, HAVING, ORDER BY, LIMIT

### WHERE 절

WHERE 절은 FROM 이후 집계 전에 행을 필터링한다. 분석에서 필요한 행만 추려내는 첫 번째 관문 역할을 하며, 조건이 복잡할수록 AND/OR를 괄호로 묶어 명확하게 표현하는 것이 중요하다.

```sql
-- 비교 연산자
WHERE salary > 50000
WHERE hire_date BETWEEN '2020-01-01' AND '2023-12-31'
WHERE department IN ('Engineering', 'Sales', 'Marketing')
WHERE name LIKE 'Kim%'          -- 대소문자 구분
WHERE name ILIKE 'kim%'         -- 대소문자 무시 (PostgreSQL/DuckDB 지원)

-- NULL 처리
WHERE manager_id IS NULL
WHERE manager_id IS NOT NULL

-- 복합 조건
WHERE salary > 50000
  AND (department = 'Engineering' OR department = 'Sales')
  AND active = true
```

### GROUP BY 절

GROUP BY는 동일한 값을 가진 행들을 하나의 그룹으로 묶어 집계 함수(COUNT, SUM, AVG 등)를 적용한다. 부서별 인원 수나 월별 매출 합계처럼 "~별 요약"이 필요한 모든 분석에 쓰인다. DuckDB는 `GROUP BY ALL`이라는 전용 문법으로 SELECT 절의 비집계 열을 자동으로 그룹화해 반복 입력을 줄여준다.

```sql
-- 단일 열 그룹화
SELECT department, COUNT(*) AS cnt
FROM employees
GROUP BY department;

-- 다중 열 그룹화
SELECT department, job_title, AVG(salary) AS avg_sal
FROM employees
GROUP BY department, job_title;

-- GROUP BY ALL (DuckDB 전용): SELECT의 비집계 열을 자동으로 그룹화
SELECT department, job_title, AVG(salary) AS avg_sal
FROM employees
GROUP BY ALL;

-- ROLLUP: 소계 포함
SELECT department, job_title, SUM(salary)
FROM employees
GROUP BY ROLLUP (department, job_title);

-- CUBE: 모든 조합의 소계
SELECT department, year, SUM(revenue)
FROM sales
GROUP BY CUBE (department, year);

-- GROUPING SETS: 원하는 그룹 조합만 선택
SELECT department, year, SUM(revenue)
FROM sales
GROUP BY GROUPING SETS (
    (department, year),
    (department),
    ()
);
```

> **팁:** `ROLLUP`과 `CUBE`는 소계와 전체 합계를 한 번의 쿼리로 구할 때 유용하다. `ROLLUP(a, b)`는 `(a, b)`, `(a)`, `()` 세 수준의 집계를 생성하고, `CUBE(a, b)`는 가능한 모든 조합 — `(a, b)`, `(a)`, `(b)`, `()` — 을 한꺼번에 만들어낸다.

### HAVING 절

HAVING은 집계 후 그룹을 필터링한다. WHERE가 개별 행을 걸러낸다면, HAVING은 GROUP BY로 만들어진 그룹 단위로 조건을 적용한다. 이 두 절의 역할 차이를 이해하면 "부서별 평균 급여가 일정 수준 이상인 부서만 추리기"와 같은 집계 후 필터링을 정확하게 작성할 수 있다.

```sql
-- 집계 결과 필터링
SELECT department, COUNT(*) AS cnt, AVG(salary) AS avg_sal
FROM employees
GROUP BY department
HAVING COUNT(*) >= 10        -- 10명 이상인 부서만
   AND AVG(salary) > 60000;  -- 평균 급여 6만 초과
```

```
WHERE vs HAVING
───────────────────────────────────────────────────
WHERE  → 집계 전 개별 행 필터링
HAVING → 집계 후 그룹 결과 필터링

-- 틀린 예: WHERE에서 집계 함수 사용 불가
WHERE COUNT(*) > 10   -- 오류!

-- 올바른 예
HAVING COUNT(*) > 10  -- 정상
```

> **주의:** WHERE 절에 집계 함수를 넣으면 DuckDB가 오류를 반환한다. 집계 결과로 조건을 걸어야 할 때는 반드시 HAVING을 사용한다. 반면 집계와 무관한 조건은 WHERE에 넣어야 DuckDB가 GROUP BY 이전 단계에서 불필요한 행을 먼저 제거해 성능을 높인다.

### ORDER BY 절

```sql
-- 오름차순 (기본값)
ORDER BY salary ASC

-- 내림차순
ORDER BY salary DESC

-- 다중 정렬: 부서 오름차순, 급여 내림차순
ORDER BY department ASC, salary DESC

-- NULL 처리: NULL을 마지막에
ORDER BY manager_id NULLS LAST

-- NULL 처리: NULL을 먼저
ORDER BY manager_id NULLS FIRST

-- 열 번호로 정렬 (열 위치 참조)
SELECT department, AVG(salary) AS avg_sal
FROM employees
GROUP BY 1
ORDER BY 2 DESC;
```

### LIMIT과 OFFSET

```sql
-- 상위 10개만
SELECT * FROM employees ORDER BY salary DESC LIMIT 10;

-- 11~20번째 행 (페이지네이션)
SELECT * FROM employees ORDER BY id LIMIT 10 OFFSET 10;

-- LIMIT 없이 OFFSET 사용 가능 (DuckDB)
SELECT * FROM employees OFFSET 100;
```

> **팁:** LIMIT은 탐색 단계에서 결과를 빠르게 확인할 때 유용하지만, ORDER BY 없이 LIMIT을 쓰면 DuckDB가 어떤 행을 반환할지 보장하지 않는다. 재현 가능한 결과가 필요하다면 ORDER BY와 함께 사용한다.
